1
Explique la arquitectura general del sistema de estacionamiento inteligente. Indique el rol de Eureka Server, API Gateway, Auth Service, User Service y al menos cuatro microservicios de dominio.

El sistema tiene una arquitectura de microservicios. Hay 12 servicios independientes que se comunican entre sí a través de la red.

Eureka Server (puerto 8761) es el registro de servicios. Cuando cada microservicio arranca, se registra en Eureka con su nombre. Así, los demás servicios pueden encontrarse por nombre sin saber la IP exacta.

API Gateway (puerto 8080) es el único punto de entrada para el cliente externo (Postman, frontend). Recibe todas las peticiones y las redirige al microservicio correspondiente usando las rutas definidas en application.yaml. Valida el token JWT antes de dejar pasar la petición.

Auth Service (8081) maneja el login. Recibe email y contraseña, verifica contra la base de datos con BCrypt, y si son válidos emite un token JWT que el cliente usará en todas las peticiones siguientes.

User Service (8082) maneja los clientes del estacionamiento: crear cliente, asignarle tipo de cliente y suscripción, actualizar datos. Tiene su propia base de datos db_usuarios.

Cuatro microservicios de dominio:

  • ms-vehiculos (8084): gestiona vehículos y tipos de vehículo. Cada vehículo pertenece a un cliente (referencia lógica).
  • ms-espacios (8085): gestiona los espacios del estacionamiento. Controla disponibilidad con un flag disponible.
  • ms-reservas (8086): permite reservar un espacio para una fecha. Valida via Feign que el cliente, vehículo y espacio existan y estén activos.
  • ms-pagos (8089): genera el cobro al finalizar un acceso. Calcula el monto con tarifa, multiplicadores y descuentos.
2
Use ms-reservas como referencia y explique el flujo Controller → Service → Repository. Indique qué responsabilidad cumple cada capa y qué DTOs participan.
ReservaController → ReservaService → ReservaRepository → MySQL db_reservas

Controller (ReservaController) recibe la petición HTTP REST. No tiene lógica de negocio — solo lee el cuerpo de la petición como un ReservaCreateDTO, llama al Service, y devuelve el ReservaResponseDTO con el código HTTP correcto (201 para crear, 200 para el resto). Usa @Valid para activar las validaciones del DTO.

Service (ReservaService) contiene toda la lógica de negocio. Llama a los Feign clients para validar cliente, vehículo y espacio. Crea la entidad Reserva, le asigna el estado inicial PENDIENTE y la guarda via Repository.

Repository (ReservaRepository) extiende JpaRepository<Reserva, Long>. Solo define las consultas personalizadas como findByEstado() y findByIdCliente(). Spring Data JPA genera el SQL automáticamente.

DTOs involucrados: ReservaCreateDTO (entrada del cliente), ReservaResponseDTO (salida), ClienteResponseDTO / VehiculoResponseDTO / EspacioResponseDTO (respuestas de los Feign clients).

3
Explique el flujo completo para crear una reserva. Debe mencionar idCliente, idVehiculo, idEspacio, validación remota mediante Feign, estado inicial PENDIENTE y persistencia.

El cliente hace POST /api/reservas enviando un JSON con idCliente, idVehiculo, idEspacio, fechaInicio y fechaFin. Eso llega al ReservaController que lo mapea a un ReservaCreateDTO.

El Controller delega al ReservaService.create(), que hace tres validaciones remotas usando Feign:

  • ClienteClient.findById(idCliente): si lanza FeignException.NotFound → excepción. Si el cliente tiene activo = false → excepción.
  • VehiculoClient.findById(idVehiculo): mismo patrón, verifica que activo = true.
  • EspacioClient.findById(idEspacio): verifica que activo = true Y disponible = true. Si el espacio está ocupado ya rechaza aquí.

Si todo pasa, crea la entidad Reserva, asigna los tres IDs como referencias lógicas (Long), las fechas, y fija el estado en PENDIENTE. Llama a reservasRepository.save(nReserva) que persiste en db_reservas. Devuelve un ReservaResponseDTO con HTTP 201.

La reserva en PENDIENTE NO bloquea el espacio todavía. El espacio se marca como no disponible solo cuando se CONFIRMA la reserva.
4
Explique cómo se confirma una reserva y qué ocurre con la disponibilidad del espacio. Mencione el cambio de estado a CONFIRMADA y la llamada remota para marcar el espacio como no disponible.

El cliente hace PUT /api/reservas/{id}/confirmar. El Controller llama a ReservaService.confirmar(id).

El Service busca la reserva en la base de datos. Valida que el estado actual sea PENDIENTE — si ya es CONFIRMADA, FINALIZADA o CANCELADA lanza una ConflictException.

Si está pendiente, cambia el estado a CONFIRMADA y hace dos cosas:

  • Llama a EspacioClient.findById(idEspacio) para obtener el espacio.
  • Llama a EspacioClient.updateDisponibilidad(espacioId, false) — esto hace un PUT /api/espacios/{id}/disponibilidad a ms-espacios que cambia el flag disponible = false.

Finalmente guarda la reserva actualizada con repository.save(reserva). A partir de este momento el espacio aparecerá como ocupado en GET /api/espacios/disponibles.

5
Explique por qué el proyecto usa referencias lógicas Long entre microservicios en vez de relaciones JPA directas entre bases de datos.

En una arquitectura de microservicios, cada servicio tiene su propia base de datos independiente. ms-reservas usa db_reservas en MySQL, y user-service usa db_usuarios, y así con los 10 servicios.

JPA y Hibernate solo pueden manejar relaciones (@ManyToOne, @JoinColumn, @ForeignKey) entre tablas dentro de la misma base de datos y el mismo contexto de persistencia. No pueden hacer joins ni foreign keys entre bases de datos separadas.

Por eso, cuando Reserva necesita referenciar a un cliente, no almacena la entidad Cliente sino un simple Long idCliente. Cuando necesita los datos del cliente, llama al ClienteClient via Feign — que hace una petición HTTP a user-service.

Esto es el patrón correcto de microservicios: la integridad referencial se valida en la lógica de negocio (el Service verifica con Feign que el cliente exista antes de guardar), no a nivel de base de datos.

Explique cómo funciona el API Gateway del proyecto. ¿Qué significa usar rutas lb://auth-service, lb://ms-reservas, lb://ms-accesos, lb://ms-pagos y lb://ms-reportes, y qué relación tiene esto con Eureka?

El API Gateway corre en el puerto 8080 y es el único punto de entrada al sistema. Todas las peticiones de Postman o del cliente van primero al Gateway.

En application.yaml, cada ruta tiene un predicates.Path (qué URL coincide) y un uri con prefijo lb://. El prefijo lb significa load balancer — le dice a Spring Cloud Gateway que no use una IP fija sino que pregunte a Eureka por el nombre del servicio.

Por ejemplo: cuando llega POST /api/reservas, el Gateway busca en su tabla de rutas, encuentra que coincide con lb://ms-reservas, pregunta a Eureka "¿dónde está registrado ms-reservas?" y Eureka responde con la IP y puerto real (localhost:8086). El Gateway redirige la petición ahí.

Esto permite que los servicios puedan cambiar de puerto o escalar horizontalmente sin cambiar la configuración del Gateway — Eureka siempre tiene el registro actualizado.

Si Eureka no está corriendo, el Gateway no puede resolver ningún nombre y todas las rutas fallan con 503.
2
Explique el flujo completo reserva → acceso → cobro → reporte. Debe indicar qué microservicio participa en cada etapa y qué información se intercambia.
ms-reservas → ms-accesos → ms-pagos → ms-reportes

1. Reserva (ms-reservas): el cliente hace POST /api/reservas con idCliente, idVehiculo, idEspacio. El servicio valida via Feign que todo existe y está activo, crea la reserva en estado PENDIENTE. Luego se confirma con PUT /api/reservas/{id}/confirmar — en ese momento el espacio se marca como no disponible en ms-espacios.

2. Entrada (ms-accesos): el guardia registra POST /api/accesos/entrada con el idReserva y la patente. AccesoService consulta a ms-reservas via Feign — solo permite entrada si la reserva está CONFIRMADA. Crea el Acceso con estado ACTIVO, registra fechaHoraEntrada = ahora y vuelve a marcar el espacio no disponible.

3. Salida (ms-accesos): PUT /api/accesos/{id}/salida. Registra fechaHoraSalida, calcula minutos = ChronoUnit.MINUTES.between(entrada, ahora), estado = COMPLETADO. Llama a ms-espacios para liberar el espacio y a ms-reservas para finalizar la reserva.

4. Cobro (ms-pagos): POST /api/cobros con idAcceso, idCliente, idMetodoPago. PagoService consulta el acceso (para obtener los minutos, idVehiculo, idEspacio), la tarifa vigente, los factores de vehículo y espacio, el multiplicador horario y los descuentos. Calcula y persiste el Cobro.

5. Reporte (ms-reportes): no tiene base de datos propia. GET /api/reportes/cobros/cliente/{id} llama a CobroClient que consulta ms-pagos. GET /api/reportes/ocupacion consulta ms-espacios. GET /api/reportes/accesos/reserva/{id} consulta ms-accesos.

3
Explique cómo se calcula un cobro en ms-pagos. Debe mencionar tarifa base, multiplicadores, tipo de vehículo, tipo de espacio, minutos y descuentos por cliente/suscripción/banco.

El PagoService.crear() recopila datos de 7 microservicios via Feign y aplica la fórmula en dos pasos:

Paso 1 — Monto base:

monto_base = precio_base_hora (tarifa vigente) × multiplicador_horario (según LABORAL / FIN_DE_SEMANA / FESTIVO) × factor_tipo_vehiculo (campo factorPrecio de TipoVehiculo) × factor_tipo_espacio (campo factorPrecio de TipoEspacio) × (minutos / 60) -- horas proporcionales

Paso 2 — Descuentos aplicados en cadena:

monto_final = monto_base × (1 - descTipoCliente / 100) × (1 - descSuscripcion / 100) × (1 - descBanco / 100)

De dónde viene cada valor:

  • precio_base_hora: TarifaClient.getTarifaVigente() → ms-tarifas
  • multiplicador_horario: HorarioTarifaClient.getVigente() → ms-tarifas (según hora actual)
  • factor_tipo_vehiculo: VehiculoClient → TipoVehiculoClient → ms-vehiculos
  • factor_tipo_espacio: EspacioClient → ms-espacios (el DTO trae el TipoEspacio embebido)
  • minutos: del campo acceso.getMinutos() calculado en el registrarSalida
  • descTipoCliente: del campo tipoCliente.descuentoPct del cliente
  • descSuscripcion: del campo suscripcion.descuentoPct si el cliente tiene suscripción activa
  • descBanco: del campo banco.descuentoPct del método de pago seleccionado

Todo se calcula con BigDecimal y RoundingMode.HALF_UP para evitar errores de redondeo en valores monetarios.

4
Explique cómo funcionan los scripts de apoyo del proyecto: load-db.ps1, set-db-port.ps1, start-all.ps1 y manage.ps1. ¿Cómo ayudan a la reproducibilidad?

load-db.ps1: crea las 10 bases de datos y carga todos los datos seed. Internamente llama al cliente MySQL con el archivo maestro db/00_run_all.sql que sourcea los 9 scripts SQL en orden. Acepta parámetros -Password y -Port para funcionar tanto con XAMPP (3306) como con Docker (3307). Esto permite que cualquier persona clone el proyecto y tenga la base de datos lista en un comando.

set-db-port.ps1: actualiza el puerto MySQL en todos los application.properties de los 12 servicios en un solo paso. Fue necesario porque en el laboratorio de la DUOC el MySQL puede estar en un puerto diferente al local. Con -DryRun muestra qué cambiaría sin aplicar nada.

start-all.ps1: lanza todos los servicios en el orden correcto de dependencias: primero Eureka, luego Gateway, luego los servicios de dominio. Cada servicio se abre en su propia ventana o pestaña de PowerShell para poder ver los logs independientemente. Acepta -Layout Tabs|Windows.

manage.ps1: dashboard interactivo para operar el sistema en vivo. Permite iniciar, detener o reiniciar servicios individualmente por número o nombre (start 4, stop all), ver estado, conectar a la base de datos. Útil durante la defensa para mostrar control.

Estos scripts son clave para la reproducibilidad: un evaluador puede clonar el repo, ejecutar load-db.ps1 y start-all.ps1, y tener el sistema corriendo sin configuración manual.
5
Explique cómo probaría el proyecto con Newman. ¿Qué valida la colección Postman de 74 requests y por qué es importante que sea idempotente?

Newman es el runner de línea de comandos de Postman. Para ejecutar la colección:

newman run estacionamientos.postman_collection.json --env-var "base=http://localhost:8080"

La colección tiene 74 requests organizados en fases que cubren el flujo completo del sistema: autenticación → clientes → vehículos → espacios → tarifas → reservas → accesos → pagos → reportes → seguridad.

Cómo funciona el token: la Fase 1 hace login a POST /api/auth/login y un script post-respuesta captura el JWT y lo guarda en una variable de entorno. Todas las fases siguientes usan ese token en el header Authorization: Bearer {token}.

Idempotente significa que puedes ejecutar la colección múltiples veces seguidas y siempre pasa, sin dejar datos sucios que rompan la siguiente ejecución. Esto se logra con pre-request scripts de limpieza: antes de crear un recurso, el script busca si ya existe y lo elimina. Así la colección siempre parte de un estado limpio.

Esto es importante porque durante la defensa se puede ejecutar en vivo varias veces para demostrar que el sistema funciona correctamente, y el evaluador puede pedirte que lo corras de nuevo sin que falle por datos duplicados.
