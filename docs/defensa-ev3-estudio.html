2.1 Los conceptos que la guía exige dominar

ConceptoQué es / qué hace
Prueba unitariaPrueba automatizada de una unidad pequeña de código (un método del Service, una regla de negocio, una respuesta de endpoint) de forma aislada, sin BD real ni servicios externos.
MockObjeto simulado que reemplaza una dependencia real (Repository, Feign Client). Permite probar el Service sin MySQL y sin que los otros microservicios estén corriendo.
@Mock vs @MockBean@Mock (Mockito puro) crea el simulado en un test aislado sin levantar Spring — es lo que usan nuestros Service tests. @MockBean/@MockitoBean reemplaza un bean dentro del contexto de Spring — lo usamos en el Controller test con @WebMvcTest.
@InjectMocksCrea la instancia real del Service bajo prueba e inyecta en ella todos los @Mock declarados.
when(...).thenReturn(...)Programa el mock: "cuando alguien llame a findById(1L), devuelve este objeto simulado".
verify(...)Comprueba que un método del mock fue llamado (o no: verify(repo, never()).save(any())). Es clave en métodos que no retornan nada, como eliminaciones o llamadas a otros servicios.
AssertsLa guía usa JUnit (assertEquals, assertNotNull). Nuestro proyecto usa AssertJ: assertThat(x).isEqualTo(y), assertThat(lista).hasSize(1), assertThatThrownBy(...).isInstanceOf(...). Cumplen el mismo rol: verificar el resultado esperado. Si el profesor pide JUnit clásico, di la equivalencia.
MockMvcSimula peticiones HTTP contra el Controller sin levantar un servidor real: mockMvc.perform(get("/api/espacios")).andExpect(status().isOk()).
jsonPath()Valida el contenido del JSON de respuesta: jsonPath("$[0].numero").value("A-01").
ObjectMapperConvierte objetos Java a JSON para el body de POST/PUT: .content(objectMapper.writeValueAsString(dto)).
Given–When–ThenEstructura del test: Given/Arrange (preparar datos y mocks) → When/Act (ejecutar el método) → Then/Assert (verificar). En nuestro código los comentarios dicen // Arrange, // Act, // Assert — es la misma convención (AAA); dilo explícitamente si te preguntan por Given–When–Then.
Diferencia Service test vs Controller test (pregunta segura): El test de Service valida lógica de negocio: reglas, cálculos, excepciones; se mockea el Repository (y los Feign Clients). El test de Controller valida el contrato HTTP: ruta, método, código de respuesta y JSON; se mockea el Service y se usa MockMvc. Un test de Controller NO valida la base de datos.

2.2 Anatomía de un test real del proyecto, línea a línea

De ms-accesos/src/test/java/.../service/AccesoServiceTest.java (este archivo existe tal cual en el repo):

@ExtendWith(MockitoExtension.class)   // activa Mockito en JUnit 5, sin levantar Spring
class AccesoServiceTest {

    @Mock private AccesoRepository accesoRepository;  // simula la BD
    @Mock private EspacioClient espacioClient;        // simula el Feign hacia ms-espacios
    @Mock private ReservaClient reservaClient;        // simula el Feign hacia ms-reservas

    @InjectMocks
    private AccesoService accesoService;              // la clase REAL bajo prueba

    @Test
    @DisplayName("registrarEntrada debe crear el acceso y marcar el espacio no disponible")
    void registrarEntrada_valido_debeCrearAccesoYBloquearEspacio() {
        // Arrange (Given): la reserva remota está CONFIRMADA y no hay acceso previo
        when(reservaClient.findById(1L)).thenReturn(reservaConfirmada);
        when(accesoRepository.findByIdReserva(1L)).thenReturn(Optional.empty());
        when(accesoRepository.save(any(Acceso.class))).thenAnswer(inv -> {
            Acceso a = inv.getArgument(0);
            a.setId(10L);                              // simula el ID que asignaría MySQL
            return a;
        });

        // Act (When): se ejecuta el método real del Service
        AccesoResponseDTO resultado = accesoService.registrarEntrada(createDTO);

        // Assert (Then): estado correcto y efecto lateral verificado
        assertThat(resultado.getId()).isEqualTo(10L);
        assertThat(resultado.getEstado()).isEqualTo("ACTIVO");
        verify(espacioClient).updateDisponibilidad(1L, false);  // ¡bloqueó el espacio!
    }
}

Qué regla de negocio valida: al registrar una entrada con reserva CONFIRMADA y sin acceso duplicado, se crea el Acceso en estado ACTIVO y además se llama a ms-espacios para marcar el espacio como no disponible. El verify() final es lo que prueba la comunicación entre microservicios sin necesitar que ms-espacios exista.

2.4 Controller test con MockMvc (el único del repo: ms-espacios)

De ms-espacios/src/test/java/.../controller/EspacioControllerTest.java. Importante conocerlo porque la guía dedica media sección a MockMvc:

@WebMvcTest(EspacioController.class)       // levanta SOLO la capa web de este controller
@AutoConfigureMockMvc(addFilters = false)  // desactiva filtros de seguridad en el test
class EspacioControllerTest {

    @Autowired private MockMvc mockMvc;
    @Autowired private ObjectMapper objectMapper;

    @MockitoBean                            // mock del Service dentro del contexto Spring
    private EspacioService espacioService;

    @Test
    void getById_inexistente_debeRetornar404() throws Exception {
        // Arrange: el service simulado lanza la excepción de "no encontrado"
        when(espacioService.findById(99L))
                .thenThrow(new ResourceNotFoundException("Espacio no encontrado con id: 99"));

        // Act + Assert: petición simulada y validación del contrato HTTP
        mockMvc.perform(get("/api/espacios/99"))
                .andExpect(status().isNotFound())                       // 404
                .andExpect(jsonPath("$.error").value("NOT_FOUND"))     // body del handler
                .andExpect(jsonPath("$.mensaje").value("Espacio no encontrado con id: 99"));
    }

    @Test
    void create_valido_debeRetornar201() throws Exception {
        when(espacioService.create(any(EspacioCreateDTO.class))).thenReturn(espacioDTO());

        mockMvc.perform(post("/api/espacios")
                        .contentType(MediaType.APPLICATION_JSON)
                        .content(objectMapper.writeValueAsString(dto)))  // objeto → JSON
                .andExpect(status().isCreated());                        // 201
    }
}
Fíjate en la cadena completa que valida el test del 404: Controller recibe GET → Service mockeado lanza ResourceNotFoundException → el GlobalExceptionHandler (@RestControllerAdvice) la captura y responde 404 con body {error: "NOT_FOUND", mensaje, timestamp}. Este test demuestra en una sola pieza: MockMvc + manejo de excepciones + códigos HTTP — exactamente lo que le faltó a Sol en EV2 (IE 2.3.4).

El test create_invalido_debeRetornar400 además valida Bean Validation: envía un DTO sin campos obligatorios y espera 400 con jsonPath("$.campos.numero").exists() — el mapa de campos que arma el handler ante MethodArgumentNotValidException.

2.5 Lo que cambia en microservicios (transición de la guía)

En el monolito Biblioteca se mockeaba solo el Repository. En microservicios también se mockea la comunicación externa. Nuestro mejor ejemplo es ReservaServiceTest (ms-reservas), que mockea 1 repository + 3 Feign Clients:

@Mock private ReservaRepository reservasRepository;
@Mock private ClienteClient clienteClient;    // → user-service
@Mock private VehiculoClient vehiculoClient;  // → ms-vehiculos
@Mock private EspacioClient espacioClient;    // → ms-espacios

// Arrange típico: los 3 servicios remotos responden OK
when(clienteClient.findById(1L)).thenReturn(new ClienteResponseDTO(1L, true));
when(vehiculoClient.findById(1L)).thenReturn(new VehiculoResponseDTO(1L, 1L, true));
when(espacioClient.findById(1L)).thenReturn(new EspacioResponseDTO(1L, "A-01", "Norte", 1, 1L, true, true));

Por qué se simula la comunicación externa (pregunta 7 de la actividad de reflexión de la guía): porque una prueba unitaria debe ser determinista y aislada — si dependiera de que user-service esté corriendo, el test fallaría por razones ajenas al código probado (red, orden de arranque, datos). El mock convierte "el otro servicio" en una respuesta controlada.

Detalle avanzado que suma puntos: PagoServiceTest usa ReflectionTestUtils.invokeMethod(...) para probar los métodos privados de la fórmula (calcularMontoBase, calcularMontoFinal) con valores exactos de BigDecimal — p. ej. valida que $1000/h × horario 2.0 × vehículo 1.5 × espacio 1.2 × 1.5h = 5400.00, y la cascada de descuentos 10%/20%/5% sobre $10.000 = 6840.00.

2.6 ⭐ Receta para escribir un test NUEVO en vivo (IE 3.1.3 — 13%)

Este es el momento de mayor puntaje de toda la defensa. El profesor pedirá algo como "agrega una prueba que valide X regla". Memoriza la plantilla y el procedimiento. Puedes anotar los pasos en papel antes de escribir.

Procedimiento (di lo que haces en voz alta):

  1. Ubicar el test existente del service correspondiente en src/test/java/.../service/ — nunca partir de un archivo vacío: copia la estructura de un @Test vecino.
  2. Identificar la regla: ¿qué método del Service? ¿caso feliz o caso de error?
  3. Arrange: programar con when(...) SOLO los mocks que ese método usa en ese camino (Mockito en modo strict falla con stubs innecesarios: UnnecessaryStubbingException).
  4. Act: llamar al método real.
  5. Assert: para caso feliz assertThat(...) + verify(...); para caso de error assertThatThrownBy(...) + verify(repo, never()).save(any()).
  6. Ejecutar: .\mvnw.cmd test -Dtest=AccesoServiceTest desde la carpeta del servicio, y mostrar el verde.

Plantilla de memoria (caso de error, el más pedido):

@Test
@DisplayName("descripción en español de la regla que valida")
void metodo_condicion_resultadoEsperado() {
    // Arrange (Given)
    when(dependenciaMock.metodo(argumento)).thenReturn(valorPreparado);

    // Act + Assert (When + Then)
    assertThatThrownBy(() -> serviceReal.metodoBajoPrueba(dto))
            .isInstanceOf(ConflictException.class)
            .hasMessageContaining("parte del mensaje");
    verify(repositoryMock, never()).save(any());
}

Ejercicio resuelto 1 — "valida que no se pueda registrar salida dos veces" (ya existe uno casi igual; así se escribiría desde cero):

@Test
@DisplayName("registrarSalida debe lanzar ConflictException si la salida ya fue registrada")
void registrarSalida_salidaYaRegistrada_debeLanzarConflict() {
    // Given: un acceso que YA tiene fechaHoraSalida
    accesoActivo.setFechaHoraSalida(LocalDateTime.now());
    when(accesoRepository.findById(1L)).thenReturn(Optional.of(accesoActivo));

    // When + Then
    assertThatThrownBy(() -> accesoService.registrarSalida(1L))
            .isInstanceOf(ConflictException.class)
            .hasMessageContaining("Salida ya registrada");
}

Línea a línea: preparo el estado inválido (salida ya seteada) → programo el repo para devolver ese acceso → ejecuto y afirmo que lanza ConflictException. No necesito mockear espacioClient porque el método falla antes de llegar a esa llamada.

Ejercicio resuelto 2 — caso feliz con verify de Feign: "al registrar salida se libera el espacio y se finaliza la reserva":

@Test
void registrarSalida_valido_liberaEspacioYFinalizaReserva() {
    // Given: acceso ACTIVO con entrada hace 1 hora (del setUp)
    when(accesoRepository.findById(1L)).thenReturn(Optional.of(accesoActivo));
    when(accesoRepository.save(any(Acceso.class))).thenAnswer(inv -> inv.getArgument(0));

    // When
    AccesoResponseDTO resultado = accesoService.registrarSalida(1L);

    // Then: minutos calculados, estado COMPLETADO, y efectos remotos verificados
    assertThat(resultado.getEstado()).isEqualTo("COMPLETADO");
    assertThat(resultado.getMinutos()).isGreaterThan(0L);
    verify(espacioClient).updateDisponibilidad(1L, true);   // libera el espacio
    verify(reservaClient).finalizarReserva(1L);             // cierra la reserva
}

Ejercicio resuelto 3 — si pide un test en otro servicio (patrón findById inexistente, sirve para CUALQUIER service del repo):

@Test
@DisplayName("findById debe lanzar NotFoundException cuando el id no existe")
void findById_inexistente_lanzaNotFound() {
    when(xxxRepository.findById(99L)).thenReturn(Optional.empty());

    assertThatThrownBy(() -> xxxService.findById(99L))
            .isInstanceOf(NotFoundException.class);
}
Ojo con el nombre de la excepción por servicio: ms-accesos y ms-reservas usan NotFoundException / ConflictException; ms-espacios y ms-pagos usan ResourceNotFoundException; ms-pagos además tiene BusinessException (ej.: "Ya existe un cobro" para el UNIQUE 1:1 con acceso). Antes de escribir, mira los imports del test vecino.

3.2 El YAML del Gateway (IE 3.3.5 — 4%; cítalo con confianza, es real)

server:
  port: 8080                    # puerto único de entrada
spring:
  application:
    name: api-gateway           # nombre con el que se registra en Eureka
  cloud:
    gateway:
      discovery:
        locator:
          enabled: true         # descubre servicios desde Eureka
      routes:
        - id: ms-accesos
          uri: lb://ms-accesos              # lb = load-balanced: resuelve por Eureka
          predicates:
            - Path=/api/accesos/**          # predicado: qué requests entran a esta ruta
        - id: ms-pagos
          uri: lb://ms-pagos
          predicates:
            - Path=/api/cobros/**,/api/bancos/**,/api/metodos-pago/**,/api/tipo-tarjetas/**
eureka:
  client:
    service-url:
      defaultZone: http://localhost:8761/eureka/   # dónde está Eureka
jwt:
  secret: ...                   # clave Base64 ≥256 bits para validar tokens

Cómo explicarlo: lb://ms-accesos significa que el Gateway no conoce IP ni puerto: le pregunta a Eureka "¿dónde vive ms-accesos?" y balancea entre las instancias registradas. El predicate Path decide qué peticiones van a cada servicio. Además el Gateway aplica el JwtAuthFilter: toda ruta excepto /api/auth/** y Swagger exige un token válido. En el test los YAML/properties cambian a H2 + eureka.client.enabled=false; en Docker, localhost se reemplaza por nombres de servicio (eureka-server, mysql-db) — esa es la diferencia local vs remoto que pide la rúbrica.

3.3 Flujo de negocio completo: reserva → acceso → cobro → reporte

Fig. 2 — Flujo secuencial: cada etapa consulta la anterior por Feign; ninguna tiene acceso directo a la base de datos de otra.

  1. Reserva (ms-reservas): create() valida por Feign que el cliente esté activo (user-service), el vehículo activo (ms-vehiculos) y el espacio activo+disponible (ms-espacios) → guarda en PENDIENTE. Máquina de estados: PENDIENTE → CONFIRMADA (bloquea el espacio vía Feign) o CANCELADA (no toca el espacio, nunca se bloqueó); CONFIRMADA → FINALIZADA.
  2. Acceso (ms-accesos): registrarEntrada valida la reserva CONFIRMADA (ReservaClient), evita duplicados, crea Acceso ACTIVO y marca el espacio no disponible (EspacioClient). registrarSalida calcula minutos, pasa a COMPLETADO, libera el espacio y finaliza la reserva. idReserva es nullable: se permite acceso sin reserva.
  3. Cobro (ms-pagos): con el acceso COMPLETADO, consulta por Feign: acceso, tarifa vigente, horario vigente, tipo de vehículo, tipo de espacio, cliente/suscripción y banco → aplica la fórmula (abajo). UNIQUE en id_acceso_ref: 1 cobro por acceso.
  4. Reporte (ms-reportes): sin BD propia; agrega datos vía Feign (ocupación de espacios, accesos por reserva, cobros por cliente). Solo lectura.

Fig. 3 — Desde PENDIENTE solo se puede confirmar o cancelar; desde CONFIRMADA solo se puede finalizar (vía ms-accesos al registrar la salida).

3.4 Fórmula de cobro (dila de memoria)

monto_base  = precio_base_hora × multiplicador_horario
              × factor_tipo_vehiculo × factor_tipo_espacio
              × (minutos / 60)

monto_final = monto_base × (1 − desc_tipo_cliente/100)
                         × (1 − desc_suscripcion/100)
                         × (1 − desc_banco/100)        ← descuentos EN CASCADA, no sumados

Todo con BigDecimal y RoundingMode.HALF_UP (nunca double para dinero). Ejemplo verificado por test: base $10.000 con descuentos 10%, 20% y 5% → 10000 × 0.90 × 0.80 × 0.95 = $6.840,00.

3.5 Mapa Feign y referencias lógicas

ms-reservas  → EspacioClient, VehiculoClient, ClienteClient
ms-accesos   → EspacioClient, ReservaClient
ms-pagos     → AccesoClient, TarifaClient, ClienteClient, VehiculoClient,
               EspacioClient, HorarioTarifaClient, TipoVehiculoClient
ms-reportes  → AccesoClient, CobroClient, EspacioClient, VehiculoClient
  • El name del @FeignClient debe calzar exacto con el spring.application.name del servicio destino — así Eureka lo resuelve.
  • Referencias lógicas: entre bases NO hay FK físicas; se guardan Long planos (@Column(name="id_espacio_ref")). La integridad se garantiza en la capa de servicio, validando por Feign antes de persistir. Justificación: cada microservicio es dueño de su BD (database-per-service); una FK cruzada acoplaría los esquemas y rompería el despliegue independiente.
  • Solo @PutMapping, nunca @PatchMapping: el HttpURLConnection por defecto de Feign no soporta PATCH; todas las actualizaciones parciales (cancelar, confirmar, finalizar, disponibilidad, salida) son PUT.
  • Feign devuelve los 404 remotos como FeignException (no como null) — por eso los services capturan y traducen la excepción.

1. Arquitectura general: rol de Eureka, Gateway, Auth, User y 4 microservicios de dominio.

"El sistema tiene 12 servicios. Eureka (8761) es el registro: cada servicio se inscribe con su spring.application.name y así se descubren entre sí sin IPs fijas. El API Gateway (8080) es el punto único de entrada: enruta con lb:// según el path y valida el JWT antes de dejar pasar. auth-service emite los tokens (login por email, BCrypt); user-service administra clientes, tipos y suscripciones. De dominio: ms-espacios (disponibilidad física), ms-reservas (máquina de estados de la reserva), ms-accesos (entrada/salida y minutos) y ms-pagos (fórmula de cobro con descuentos). Cada uno con su propia base MySQL — database per service."

2. Flujo Controller → Service → Repository en ms-reservas, con DTOs.

"El Controller solo orquesta HTTP: recibe el JSON como ReservaCreateDTO (con Bean Validation), llama al service y devuelve ReservaResponseDTO en un ResponseEntity — nunca expone la entidad JPA como contrato. El Service (@Service @Transactional) concentra la lógica: validar cliente/vehículo/espacio por Feign, aplicar la máquina de estados, loguear con @Slf4j. El Repository (JpaRepository<Reserva, Long>) persiste. Las excepciones suben al GlobalExceptionHandler. Ventaja de los DTOs: desacoplan el contrato REST del modelo de datos — puedo cambiar la entidad sin romper a los consumidores."

3. Crear una reserva: idCliente, idVehiculo, idEspacio, validación Feign, PENDIENTE, persistencia.

"create() recibe los 3 ids + fechas. Antes de guardar valida remotamente: cliente existe y activo=true (ClienteClient → user-service), vehículo activo (VehiculoClient), espacio activo y disponible (EspacioClient). Si algo falla, Feign lanza excepción (un 404 remoto llega como FeignException, no como null) y se traduce a NotFound/Conflict — la reserva NO se guarda. Si todo pasa, se persiste con estado=PENDIENTE y los ids como referencias lógicas Long. Importante: crear la reserva no bloquea el espacio todavía."

4. Confirmar una reserva y qué pasa con la disponibilidad.

"confirmar() valida la transición (solo desde PENDIENTE), cambia a CONFIRMADA y ahí sí llama por Feign a PUT /api/espacios/{id}/disponibilidad?disponible=false para bloquear el espacio. Por eso cancelar() no toca ms-espacios: en PENDIENTE el espacio nunca se bloqueó. Y la liberación ocurre en ms-accesos al registrar la salida (disponible=true de vuelta)."

5. ¿Por qué referencias lógicas Long en vez de relaciones JPA entre bases?

"Porque cada microservicio es dueño exclusivo de su base (10 BDs separadas). Una @ManyToOne entre servicios exigiría que ambas tablas vivan en el mismo esquema — eso acopla los despliegues y rompe la autonomía. Guardamos Long planos (id_espacio_ref) y garantizamos la integridad en la capa de servicio validando por Feign antes de persistir. El trade-off es consistencia eventual: lo aceptamos a cambio de servicios independientes. Dentro de un mismo servicio sí usamos JPA real: Vehiculo → TipoVehiculo es @ManyToOne porque ambas tablas están en db_vehiculos."

6.3 Preguntas nuevas estilo EV3 (testing, Swagger, YAML, Docker)

¿Qué diferencia hay entre probar un Service y probar un Controller? (reflexión #3 de la guía)

"El Service test valida lógica de negocio con Mockito puro — mockeo Repository y Feign Clients, y afirmo sobre resultados y excepciones. El Controller test valida el contrato HTTP con @WebMvcTest + MockMvc — mockeo el Service y afirmo sobre ruta, código de estado y JSON (jsonPath). Errores que detecta el de Controller: ruta mal mapeada, código HTTP incorrecto, JSON con campos distintos, validación que no dispara. Errores que detecta el de Service: regla de negocio mal implementada, cálculo erróneo, excepción que no se lanza."

¿Por qué las pruebas no deben usar la base de datos real? (reflexión #1)

"Tres razones: aislamiento (un test no debe depender del estado que dejó otro test o el desarrollo manual), velocidad y reproducibilidad (mismo resultado en cualquier máquina, incluso sin MySQL). Por eso los Service tests mockean el Repository, y los tests de integración usan H2 en memoria con ddl-auto=create-drop en application-test.properties, con Eureka deshabilitado."

¿Qué hace exactamente verify() y cuándo es imprescindible?

"Comprueba interacciones, no valores: que el service llamó (o no llamó) a un método del mock. Es imprescindible cuando el efecto no está en el valor de retorno: en registrarSalida, el retorno es el DTO, pero la liberación del espacio es una llamada Feign — solo verify(espacioClient).updateDisponibilidad(1L, true) puede probarla. Y verify(repo, never()).save(any()) prueba que ante un error NO se persistió nada."

Muéstrame en Swagger cómo se consume un endpoint.

Abrir :8085/swagger-ui/index.html → POST /api/espacios → mostrar el schema de EspacioCreateDTO (campos y obligatorios) → "Try it out" con un body válido → 201 y el response DTO → repetir con body inválido → 400 con el mapa campos. Cerrar con: "la doc se genera del código con springdoc, por eso nunca se desalinea del comportamiento real".

¿Qué pasa si corro los tests y aparece No qualifying bean of type EspacioService?

"Es el error clásico de @WebMvcTest sin mock: ese slice solo levanta la capa web, no los services reales. El Controller necesita su dependencia, así que hay que declararla como @MockitoBean. Está en la sección 11.6 de la guía de testing."

¿Por qué en Docker el servicio no encuentra a MySQL si en local funciona?

"Porque dentro de un contenedor localhost apunta al propio contenedor. En la red de docker compose los servicios se resuelven por nombre: la URL pasa de jdbc:mysql://localhost:3306/db_accesos a jdbc:mysql://mysql-db:3306/db_accesos, y Eureka de localhost:8761 a eureka-server:8761. Es el error #1 de la guía Docker del profesor."

¿Cómo garantizan que un acceso tenga máximo un cobro?

"Doble capa: constraint UNIQUE en cobro.id_acceso_ref (BD) y validación en PagoService.crear() que consulta findByIdAcceso y lanza BusinessException ('Ya existe un cobro') → 422. Hay un test unitario que lo prueba con el repo mockeado."

¿Por qué usan @PutMapping para cancelar/confirmar en vez de @PatchMapping?

"Decisión técnica documentada: el cliente HTTP por defecto de Feign (HttpURLConnection) no soporta el verbo PATCH, así que todas las actualizaciones parciales entre servicios son PUT. Además fue el patrón enseñado en clases."
