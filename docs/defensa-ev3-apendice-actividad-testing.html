Actividad 1 — Análisis de Service Test

La guía pide seleccionar una prueba de Service y completar la tabla. Resolvemos dos casos: uno con comunicación entre microservicios (ms-accesos) y uno de servicio con lógica propia (ms-espacios), para cubrir ambos perfiles de pregunta.

Caso A — ms-accesos: entrada válida crea el acceso y bloquea el espacio

Test analizado (existe tal cual en el repo):

@Test
@DisplayName("registrarEntrada debe crear el acceso y marcar el espacio no disponible")
void registrarEntrada_valido_debeCrearAccesoYBloquearEspacio() {
    // Arrange
    when(reservaClient.findById(1L)).thenReturn(reservaConfirmada);
    when(accesoRepository.findByIdReserva(1L)).thenReturn(Optional.empty());
    when(accesoRepository.save(any(Acceso.class))).thenAnswer(inv -> {
        Acceso a = inv.getArgument(0);
        a.setId(10L);
        return a;
    });

    // Act
    AccesoResponseDTO resultado = accesoService.registrarEntrada(createDTO);

    // Assert
    assertThat(resultado.getId()).isEqualTo(10L);
    assertThat(resultado.getEstado()).isEqualTo("ACTIVO");
    verify(espacioClient).updateDisponibilidad(1L, false);
}

Caso B — ms-espacios: número de espacio duplicado rechaza la creación

Test analizado:

@Test
@DisplayName("create debe lanzar IllegalArgumentException cuando el número está duplicado")
void create_numeroDuplicado_debeLanzarExcepcion() {
    // Arrange
    EspacioCreateDTO dto = new EspacioCreateDTO("A-01", "Norte", 1, 1L, true, true);
    when(espaciosRepository.existsByNumero("A-01")).thenReturn(true);

    // Act + Assert
    assertThatThrownBy(() -> espacioService.create(dto))
            .isInstanceOf(IllegalArgumentException.class)
            .hasMessageContaining("A-01");
    verify(espaciosRepository, never()).save(any(Espacio.class));
}

Cómo se aplicaría en DoggySpaEs el patrón de la Etapa 1 (ms-cliente simple, sin dependencias remotas): regla de unicidad probada solo contra el Repository simulado, sin Feign ni base de datos real
Contraste entre ambos casos (esto es lo que la guía quiere que notes): el Caso B es el testing "estilo Biblioteca" (Service → Repository); el Caso A es el testing de microservicios (Service → Repository y Service → Clients externos). La técnica es idéntica — when/thenReturn, asserts, verify — lo que cambia es cuántas fronteras se simulan.

Actividad 2 — Análisis de Controller Test

Test seleccionado: el único Controller test del repo, con MockMvc. Analizamos el caso 404, que además demuestra el manejo de excepciones y códigos HTTP.

@WebMvcTest(EspacioController.class)
@AutoConfigureMockMvc(addFilters = false)
class EspacioControllerTest {

    @Autowired private MockMvc mockMvc;
    @MockitoBean private EspacioService espacioService;

    @Test
    @DisplayName("GET /api/espacios/{id} debe retornar 404 cuando el espacio no existe")
    void getById_inexistente_debeRetornar404() throws Exception {
        // Arrange
        when(espacioService.findById(99L))
                .thenThrow(new ResourceNotFoundException("Espacio no encontrado con id: 99"));

        // Act + Assert
        mockMvc.perform(get("/api/espacios/99"))
                .andExpect(status().isNotFound())
                .andExpect(jsonPath("$.error").value("NOT_FOUND"))
                .andExpect(jsonPath("$.mensaje").value("Espacio no encontrado con id: 99"));
    }
}

Posible causa de errorRuta que no calza con el @RequestMapping (→ 404 falso); esperar isOk() cuando el endpoint devuelve 201; campo JSON con otro nombre ($.error vs $.status — ¡los handlers varían entre servicios!); olvidar el @MockitoBean → No qualifying bean of type EspacioService
Cómo se aplicaría en DoggySpaIdéntico: el Controller test no cambia al pasar a microservicios porque el Service está simulado. En ms-cliente de DoggySpa sería @WebMvcTest(ClienteController.class) + @MockitoBean ClienteService + MockMvc validando ruta, código y JSON
Lo que este test demuestra en una sola pieza (y que vale puntos en IE 2.3.4 / 3.1.2): la cadena completa Controller recibe GET → Service lanza excepción → @RestControllerAdvice la captura → ResponseEntity 404 con body {error, mensaje, timestamp}. Es la prueba de que el manejo de errores no es teórico: está testeado.

18. Actividad de reflexión — respondida en equipo

1. ¿Por qué no es recomendable probar directamente con la base de datos de desarrollo?

Porque el test deja de ser aislado y determinista: dependería del estado que dejaron corridas anteriores o el trabajo manual, sería lento, y podría corromper datos que estamos usando para desarrollar. Además fallaría por razones ajenas al código (MySQL apagado, puerto cambiado). Por eso nuestros Service tests mockean el Repository, y los de integración usan H2 en memoria con ddl-auto=create-drop en application-test.properties, con Eureka deshabilitado (eureka.client.enabled=false).

2. ¿Qué ventaja tiene usar mocks?

Permiten probar una clase sin sus dependencias reales: sin base de datos, sin red y sin que los otros microservicios estén corriendo. El mock devuelve exactamente lo que el escenario necesita (when().thenReturn()), incluso casos difíciles de reproducir en real (un servicio caído, un dato inexistente), y además registra las interacciones para verificarlas (verify()). Resultado: tests rápidos, repetibles y enfocados en una sola regla.

3. ¿Qué diferencia existe entre probar un Service y probar un Controller?

El Service test valida lógica de negocio con Mockito puro (@ExtendWith(MockitoExtension.class)): reglas, cálculos, excepciones; se mockean Repository y Feign Clients. El Controller test valida el contrato HTTP con @WebMvcTest + MockMvc: ruta, método, código de respuesta y JSON; se mockea el Service. Uno responde "¿la regla está bien implementada?", el otro "¿el endpoint expone bien esa regla?".

4. ¿Qué tipo de errores detecta una prueba de Controller?

Ruta mal mapeada (404), método HTTP incorrecto, código de respuesta equivocado (200 vs 201), JSON con campos distintos a lo pactado (falla el jsonPath), validación Bean Validation que no dispara (400 ausente), y manejo de excepciones roto (el handler no traduce la excepción al código correcto).

5. ¿Qué tipo de errores detecta una prueba de Service?

Reglas de negocio mal implementadas (permitir una entrada con reserva PENDIENTE), cálculos erróneos (la fórmula de PagoService con BigDecimal: 1000×2.0×1.5×1.2×1.5h debe dar exactamente 5400.00), excepciones que no se lanzan cuando corresponde, efectos laterales ausentes (no llamar a updateDisponibilidad) y persistencias indebidas (guardar pese a un error — lo caza verify(repo, never()).save(any())).

6. ¿Qué cambia cuando un microservicio depende de otro?

Aparece una frontera nueva que también hay que simular: el cliente HTTP (Feign en nuestro caso). En ReservaServiceTest se mockean 1 repository + 3 clients (ClienteClient, VehiculoClient, EspacioClient); en AccesoServiceTest, 1 repository + 2 clients. El test además debe cubrir los caminos donde el remoto falla o devuelve datos inválidos (cliente inactivo, espacio no disponible).

7. ¿Por qué en microservicios se debe simular también la comunicación externa?

Porque una prueba unitaria debe poder correr sin que el resto del ecosistema exista. Si el test de ms-accesos necesitara ms-reservas corriendo, fallaría por red, orden de arranque o datos ajenos — dejaría de ser unitaria. Simular el client convierte "el otro servicio" en una respuesta controlada y permite probar escenarios remotos imposibles de orquestar en vivo (404 remoto, servicio caído → FeignException).

8. ¿Qué se debería validar antes de afirmar que un endpoint está correctamente probado?

Cinco cosas: (1) la ruta y método exactos; (2) el código HTTP real de cada camino (feliz y de error); (3) el JSON de respuesta campo a campo; (4) la interacción con el Service (que se llamó con los argumentos correctos); y (5) los casos de error: validación 400, recurso inexistente 404, conflicto 409. Un endpoint con solo el caso feliz probado no está correctamente probado.

19. Evidencia esperada de la actividad — checklist de cumplimiento

Lo que la guía pide entregar por equipo, y dónde queda cubierto en este apéndice:

  • Análisis de al menos un Service Test — Actividad 1, con dos casos: AccesoServiceTest (con Feign) y EspacioServiceTest (regla de unicidad).
  • Análisis de al menos un Controller Test — Actividad 2: EspacioControllerTest con MockMvc (404, y referencia a los casos 201/400/200 de la misma clase).
  • Explicación de los mocks utilizados — filas "Repository simulado" y "Service simulado" de cada tabla: @Mock/@InjectMocks en Service tests, @MockitoBean en el Controller test, y por qué en microservicios se mockean además los Feign Clients.
  • Explicación de los asserts utilizados — filas "Assert utilizado": AssertJ (assertThat, assertThatThrownBy) como equivalente de assertEquals/assertNotNull de la guía, más verify()/never() para interacciones y jsonPath() para el contrato JSON.
  • Identificación de posibles errores — filas "Posible causa de error" de las tres tablas + los errores de entorno (BD real, perfil, mock faltante) tratados en la sección 2.7 de la guía de estudio.
  • Breve relación con cómo se aplicaría en microservicios — filas "Cómo se aplicaría en DoggySpa": nuestro proyecto ya implementa las tres etapas que la guía anuncia para DoggySpa (servicio simple → servicio con lógica propia → servicio con comunicación), así que la relación se demuestra con código propio.
Nota para la defensa: la guía dice explícitamente que "no se busca solamente que el test funcione, sino que el estudiante pueda explicar qué está ocurriendo en cada parte". Este apéndice es el guion de esa explicación; la plantilla para escribir un test nuevo en vivo (IE 3.1.3, 13%) está en la sección 2.6 de la guía de estudio.
