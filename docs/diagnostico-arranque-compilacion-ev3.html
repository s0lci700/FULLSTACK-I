1. Causa raíz — por qué pasa el 429

mvnw.cmd es el Maven Wrapper. La primera vez que se ejecuta en un computador, antes de compilar cualquier cosa, necesita descargar maven-wrapper.jar desde internet. Esa descarga golpea los servidores de Maven Central.

Un 429 es un límite de peticiones por IP. En un laboratorio, decenas de estudiantes compilando en la misma ventana horaria (08:30–10:00) salen todos por la misma IP pública (NAT del establecimiento). Maven Central ve ese volumen concentrado y empieza a rechazar peticiones — a cualquiera, sin importar si el pom.xml o el código de cada estudiante están bien.

Por qué el Maven local del repo (apache-maven-3.9.15) tampoco salvó la situación

El repo trae un Maven ya instalado en apache-maven-3.9.15/bin/mvn.cmd pensado justamente para evitar el wrapper. Probé ambos modos en un equipo con caché de dependencias vacía (escenario "PC de laboratorio recién usado"):

ModoResultadoPor qué
mvn.cmd -o clean compile (offline)Falla: Non-resolvable parent POM ... in offline modeOffline solo reutiliza lo que ya está en ~/.m2/repository. Si nunca se compiló antes en ese PC, no hay nada que reutilizar.
mvn.cmd clean compile (online)Falla: status code: 429, Too Many RequestsSí llega a internet (a diferencia del wrapper, no necesita descargarse a sí mismo), pero pega contra el mismo límite de Maven Central al bajar el POM padre de Spring Boot.

Conclusión: el Maven local sí evita el punto de falla del wrapper (descargarse a sí mismo), pero solo protege del 429 si la caché .m2 ya estaba poblada antes del examen.

2. Factor secundario — versión de Java activa

El proyecto fija <java.version>21</java.version> en el pom.xml raíz (heredado de spring-boot-starter-parent:3.5.14). En el equipo donde reproduje el escenario, JAVA_HOME apuntaba por defecto a JDK 25 aunque JDK 21 también estaba instalado.

La guía oficial de Duoc para laboratorios no fuerza una versión concreta de JDK (su propio ejemplo muestra java version "25"), así que este no es necesariamente el problema principal — pero si el PC del laboratorio tenía un JDK muy nuevo sin configurar explícitamente, es un segundo punto de falla posible (Lombok y algunos plugins tardan en soportar versiones de JDK recién salidas). No se pudo confirmar si aplicó el día de la defensa porque el build nunca pasó del 429.

3. La solución — ya estaba escrita en el repo

Del documento docs/Instalación Maven en pc laboratorios duoc.docx, sección "Problema 2":

Regla de oro para PCs de laboratorio No usar .\mvnw.cmd. Usar el Maven instalado de forma independiente (mvn), porque no depende de descargarse a sí mismo cada vez.

Checklist para la próxima vez

  • Antes del examen (con internet, en cualquier PC, con anticipación): correr una vez mvn clean install -DskipTests en cada servicio para poblar ~/.m2/repository. Esa caché es reutilizable luego en modo offline.
  • Instalar Maven como binario independiente (MAVEN_HOME apuntando a C:\apache-maven-3.9.15, agregado al Path como %MAVEN_HOME%\bin) en vez de depender del wrapper — instrucciones completas ya en docs/Instalación Maven en pc laboratorios duoc.docx.
  • Verificar JAVA_HOME explícitamente antes de compilar: java -version y javac -version deben mostrar la misma versión que espera el pom.xml (21). Si el equipo tiene varios JDK, fijar JAVA_HOME a mano en vez de confiar en el que quedó por defecto.
  • Nunca ejecutar .\mvnw.cmd en un PC de laboratorio compartido — usar mvn clean install -DskipTests y luego mvn spring-boot:run directamente.
  • Si aparece mvn no se reconoce como un comando: revisar que MAVEN_HOME exista y que el Path tenga %MAVEN_HOME%\bin, luego cerrar y reabrir VS Code (las variables de entorno solo se leen al reiniciar la terminal/IDE).
  • Si aparece error de dependencias incompletas o corruptas: mvn clean install -U -DskipTests (el -U fuerza revisar de nuevo las dependencias remotas).
  • Orden de arranque real una vez compilado: eureka-server primero, esperar que registre, luego los microservicios de negocio, api-gateway al final (Gateway depende de que Eureka ya tenga registrados los servicios que va a enrutar).

4. Flujo recomendado, paso a paso

# 1. Verificar herramientas
java -version
mvn -version

# 2. Entrar al servicio
cd eureka-server

# 3. Compilar SIN el wrapper
mvn clean install -DskipTests

# 4. Ejecutar
mvn spring-boot:run

# 5. Repetir 2-4 para cada microservicio, en el orden:
#    eureka-server -> auth/user/security/vehiculos/espacios/tarifas -> reservas/accesos -> pagos -> reportes -> api-gateway
