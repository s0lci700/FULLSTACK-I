Track A — HTTP Status Codes
✓SeveridadServicioArchivoHallazgo
LOW auth-service AuthController.java POST /api/auth/login retorna 200. Semánticamente aceptable para auth, pero el resto del proyecto usa 201 para creación. Inconsistencia menor de convención.
Track B — Feign / Route Mismatches
✓SeveridadServicioArchivoHallazgo
HIGH api-gateway application.yaml La ruta id: security-service define Path=/api/permisos/**,/api/rol-permisos/**, pero RolPermisoController está mapeado a /api/roles-permisos (con 's'). Las peticiones a /api/roles-permisos/** pasadas por el gateway retornarán 404. La ruta correcta en la gateway debería ser /api/roles-permisos/**.
Track C — DTO Symmetry (productor / consumidor)
✓SeveridadServicioArchivoHallazgo
HIGH ms-reportes dto/CobroResponseDTO.java El campo local se llama fechaHoraCobro (tipo LocalDateTime), pero ms-pagos serializa el campo como fechaCobro (tipo String). La deserialización Feign fallará en runtime al llamar a GET /api/cobros/cliente/{id} — Jackson no podrá mapear el campo por nombre ni por tipo.
MED ms-pagos dto/CobroResponseDTO.java El campo fechaCobro está tipado como String y asignado con .toString() en el mapper. Pierde información de tipo; debería ser LocalDateTime con serialización via @JsonFormat. Este cambio en conjunto con el fix del DTO de ms-reportes.
Track D — GlobalExceptionHandler
✓SeveridadServicioArchivoHallazgo
HIGH ms-pagos exception/GlobalExceptionHandler.java El cuerpo de respuesta es inconsistente con el patrón del proyecto: ResourceNotFoundException retorna solo {error}, BusinessException retorna solo {error}, MethodArgumentNotValidException retorna solo el mapa de campos. Ninguno incluye timestamp ni mensaje. El patrón estándar del proyecto (definido en CLAUDE.md) requiere error + mensaje + timestamp en todas las respuestas.
MED user-service exception/GlobalExceptionHandler.java El campo timestamp está ausente en la mayoría de respuestas de error. El handler retorna Map<String,String> en vez de Map<String,Object> (timestamp requiere Object para guardar el valor de LocalDateTime).
MED security-service exception/GlobalExceptionHandler.java El campo timestamp está ausente en todas las respuestas de error. Mismo problema que user-service: retorna Map<String,String>.

MED user-service application.properties ddl-auto=update — debería ser validate.

Track G — Soft Delete
✓SeveridadServicioArchivoHallazgo
MED ms-espacios service/EspacioService.java El método delete() llama a repository.deleteById(id) — hard delete. El patrón de soft delete del proyecto (usado en ms-vehiculos y user-service) requiere marcar activo=false en lugar de eliminar la fila. Un espacio eliminado hard podría romper referencias en ms-reservas y ms-accesos.
Track H — Security
⚠ Todos los hallazgos de este track son HIGH. La ausencia de validación JWT en la gateway y en los servicios expone todos los endpoints sin autenticación.
✓SeveridadServicioArchivoHallazgo
HIGH api-gateway security/JwtAuthFilter.java JwtAuthFilter es un stub vacío — no extrae, valida ni reenvía tokens JWT. No hay autenticación aplicada a nivel de gateway. Cualquier petición sin token pasa al servicio destino.
HIGH user-service SecurityConfig.java Archivo SecurityConfig.java ausente — todos los endpoints del servicio son públicamente accesibles sin JWT.
HIGH ms-vehiculos SecurityConfig.java Archivo SecurityConfig.java ausente — endpoints sin protección JWT.
HIGH ms-espacios SecurityConfig.java Archivo SecurityConfig.java ausente — endpoints sin protección JWT.
HIGH ms-reservas SecurityConfig.java Archivo SecurityConfig.java ausente — endpoints sin protección JWT.
HIGH security-service SecurityConfig.java Archivo SecurityConfig.java ausente — endpoints sin protección JWT.
Nota: auth-service sí tiene SecurityConfig.java completo con BCrypt + JWT. Los otros servicios deben validar el token generado por auth-service usando JwtUtil compartido y un JwtAuthFilter equivalente.
Track I — Business Logic / Code Quality
✓SeveridadServicioArchivoHallazgo
MED ms-accesos service/AccesoService.java Línea ~37: if (reserva == null) es dead code — Feign lanza FeignException ante un 404, nunca retorna null. La verificación de null nunca se ejecuta en producción y da falsa sensación de seguridad.
MED ms-reservas service/ReservaService.java Líneas 63–71 aprox.: null checks en las tres respuestas Feign (espacio == null, vehiculo == null, cliente == null) son dead code por la misma razón.
LOW ms-vehiculos service/VehiculoService.java listarTodos() lanza NotFoundException cuando la lista está vacía. El comportamiento correcto para una colección vacía es retornar [] (HTTP 200), no 404.
LOW ms-espacios service/EspacioService.java findAll() y findDisponibles() lanzan NotFoundException cuando la lista está vacía. Mismo problema que ms-vehiculos.
LOW ms-reportes client/VehiculoClient.java VehiculoClient está definido pero nunca es inyectado ni llamado en ReporteService. Dead code — el cliente puede eliminarse o implementarse.
LOW ms-pagos service/PagoService.java La falla de horarioTarifaClient.getVigente() es capturada silenciosamente con fallback a 1.0 (el servicio sí loguea WARN). El comportamiento es aceptable pero podría ocultar errores de configuración de horarios.
