Setup antes de grabar

Ejecutar estos comandos en orden antes de iniciar la grabación. El orden importa.

# 1. Resetear bases de datos con seed data limpio
.\load-db.ps1

# 2. Levantar todos los servicios en orden correcto
.\start-all.ps1

# 3. Verificar que Eureka muestre todos los servicios en UP
#    http://localhost:8761

# 4. Abrir Postman y borrar variables: token, reservaId2, accesoId
#    (Edit Collection → Variables → limpiar valores)

# 5. Verificar Newman antes de grabar (opcional pero recomendado)
npx newman run estacionamiento.postman_collection.json --env-var "base=http://localhost:8080"

Arquitectura del Sistema
0:40 → 1:40 · 45–60 seg

Abrir docs/ARQUITECTURA.html o mostrar el diagrama de microservicios mientras se explica.

Guión sugerido "El sistema implementa arquitectura de microservicios con Spring Boot 3.5 y Java 21. Tenemos 12 servicios independientes: Eureka Server en el puerto 8761 actúa como registro de servicios, el API Gateway en 8080 es el único punto de entrada y valida los tokens JWT, y los 10 microservicios de dominio tienen cada uno su propia base de datos MySQL — no hay tablas compartidas."

"La comunicación entre servicios usa Feign Client. Por ejemplo, cuando ms-pagos calcula un cobro, llama a ms-accesos para obtener los minutos, a ms-tarifas para el precio base, a ms-espacios para el factor de tipo espacio, y a user-service para el descuento del cliente — todo en una sola operación."
  • 12 microservicios implementados y su responsabilidad
  • API Gateway (puerto 8080) — punto de entrada único, valida JWT
  • Eureka Server (puerto 8761) — service discovery
  • Comunicación Feign Client (sincrónica) entre servicios
  • 10 bases de datos separadas, una por microservicio de dominio

Demostración Funcional en Postman
2:40 → 5:40 · ~3 min

Parte A — Happy path (~90 seg)

Ejecutar en orden. Mantener la consola de logs visible a la derecha para que se vean los logs SLF4J al hacer el cobro.

RequestMétodo + EndpointQué mostrar / decir
#1 POST /api/auth/login Token JWT en el body de respuesta — se guarda en la variable token automáticamente
#44 POST /api/reservas 201 Created, el sistema valida cliente activo + vehículo + espacio disponible vía Feign
#47b PUT /api/reservas/{id}/confirmar Estado PENDIENTE → CONFIRMADA (máquina de estados)
#51 POST /api/accesos/entrada 201 — ver en logs cómo ms-accesos llama a ms-espacios para marcar el espacio ocupado
#52 PUT /api/accesos/{id}/salida 200 — campo minutos calculado, espacio liberado, estado COMPLETADO
#56 POST /api/cobros 201 — montoFinal calculado con BigDecimal. ms-pagos llama a 4 servicios vía Feign para obtener todos los factores de la fórmula
Qué decir en el cobro

Mostrar el montoFinal en el response y explicar: "ms-pagos consulta el precio base a ms-tarifas, el multiplicador horario, el factor de tipo de vehículo, el factor de tipo de espacio, y los descuentos del cliente y banco — todo desde microservicios independientes. El cálculo se hace con BigDecimal para precisión monetaria."

Parte B — Casos de error (~60 seg)

CasoCómo provocarloRespuesta esperada
Validación fallida POST /api/clientes — eliminar el campo rut del body antes de enviar 400 con {"campos": {"rut": "no debe estar vacío"}}
Recurso no existe GET /api/clientes/999 404 Not Found — GlobalExceptionHandler, body estándar con error, mensaje, timestamp
Regla de negocio POST /api/cobros — mismo accesoId por segunda vez (request #57) 422 Unprocessable Entity — UNIQUE constraint en id_acceso_ref, un acceso solo puede tener un cobro
Qué decir en los errores "Todos los errores pasan por GlobalExceptionHandler. El formato de respuesta es siempre el mismo en los 12 servicios: código HTTP correcto, campo error, campo mensaje legible, y timestamp. Los errores de validación incluyen además un mapa campos con el nombre del campo y por qué falló."

Restricciones críticas

Sin commits después de entregar

Cualquier commit en GitHub detectado después de la entrega del martes 19/05 y antes de la defensa del viernes 22/05 resulta en nota 1.0 automática, grupal e individual. Una vez subido el video, no hacer ningún commit hasta después del viernes.

Puerto de MySQL en el laboratorio

Las máquinas del laboratorio usan MySQL en el puerto 3307, no 3306. Antes de la defensa del viernes, ejecutar:

.\set-db-port.ps1 -Port 3307
Postman debe estar listo para la defensa

La colección ya está importada en la nube. Para la defensa presencial del viernes: importar el archivo local estacionamiento.postman_collection.json en Postman sin conexión a internet.
