1. Contexto y metodología

El sistema de estacionamiento inteligente consta de 12 microservicios Spring Boot independientes que se comunican a través de un API Gateway (puerto 8080). Una colección Postman de 74 requests cubre el flujo completo de extremo a extremo: autenticación → gestión de datos maestros → operación de estacionamiento → facturación → reportes.

La sesión de depuración usó Newman (CLI de Postman) ejecutado directamente desde el asistente para obtener resultados reproducibles sin depender de la interfaz gráfica de Postman. Cada ciclo consistió en:

Ejecución Newman
Se ejecuta la suite completa y se observan los assertions que fallan.
Diagnóstico de la causa raíz
Se leen los logs de respuesta HTTP, se cruzan con el código fuente Java y el esquema de BD.
Corrección y recompilación
.\mvnw.cmd clean package -DskipTests — BUILD SUCCESS requerido antes de continuar.
Reinicio del servicio y reverificación
El servicio carga el nuevo JAR y se repite el ciclo Newman.

En paralelo se corrigió la propia colección Postman cuando el error era del lado del cliente (cuerpo incorrecto, método HTTP, variable no capturada).

4. Bugs encontrados y corregidos

B-01

Token JWT no capturado — todos los requests autenticados retornan 401

Colección Postman
Síntoma Requests de Fase 2 en adelante reciben 401 Unauthorized en lugar del código esperado. Causa raíz El script de test del login no extraía el token de la respuesta y tampoco se enviaba el header Authorization en los requests siguientes. La variable {{token}} nunca tenía valor. Fix Se agregó al test-script del login:
pm.collectionVariables.set('token', pm.response.json().token);
Y se añadió el header Authorization: Bearer {{token}} a los 74 requests que requieren autenticación. Impacto Sin este fix, todos los requests de Fase 2–12 fallan. Es el bloqueo de mayor impacto de la sesión.
B-02

Método HTTP PATCH no soportado por Feign — error de comunicación entre servicios

Feign / Spring
Síntoma Los endpoints de ms-accesos y ms-reservas que invocan a ms-espacios via Feign lanzaban feign.RetryableException: Invalid HTTP method: PATCH. Causa raíz El cliente Feign por defecto usa java.net.HttpURLConnection, que no incluye PATCH en los métodos HTTP permitidos. Los controladores usaban @PatchMapping y los clientes Feign declaraban @PatchMapping en la interfaz. Decisión El docente solo había enseñado PUT (no PATCH), por lo que se decidió reemplazar todos los @PatchMapping por @PutMapping en lugar de agregar la dependencia Apache HC5. Esto simplifica el stack y es consistente con el currículo del curso. Archivos ms-espacios/EspacioController.java · ms-reservas/ReservaController.java · ms-accesos/AccesoController.java · ms-reservas/client/EspacioClient.java · ms-accesos/client/EspacioClient.java · ms-accesos/client/ReservaClient.java
B-03

findByActivoTrue() lanza excepción cuando hay múltiples tarifas activas

ms-tarifas
Síntoma GET /api/tarifas/vigente → 500 Internal Server Error. Causa raíz El método del repositorio era Optional<Tarifas> findByActivoTrue(). Spring Data JPA lanza IncorrectResultSizeDataAccessException cuando la consulta devuelve más de una fila. Los datos de seed insertaban dos tarifas con activo = true (TARIFA_BASE y TARIFA_NOCTURNA). Fix Se cambió a findFirstByActivoTrue(), que agrega un LIMIT 1 implícito y devuelve la primera tarifa activa sin lanzar excepción. 
- Optional<Tarifas> findByActivoTrue();
+ Optional<Tarifas> findFirstByActivoTrue();
B-04

ClienteCreateDTO no incluye el campo rut — violación de constraint NOT NULL

user-service
Síntoma POST /api/clientes → 500 Internal Server Error. Causa raíz La entidad Cliente define rut VARCHAR NOT NULL UNIQUE en la base de datos, pero el DTO ClienteCreateDTO no tenía el campo rut. El servicio nunca llamaba cliente.setRut(), por lo que MySQL rechazaba el insert con una violación de constraint. Fix Se agregó @NotBlank private String rut; al DTO y cliente.setRut(dto.getRut()); en el método create() del servicio.
B-05

EspacioService.delete() modifica el DTO en lugar de la entidad — soft-delete silencioso

ms-espacios
Síntoma DELETE /api/espacios/{id} devolvía 204 No Content (aparentemente exitoso), pero el espacio seguía apareciendo como activo al consultarlo. Causa raíz El método delete() obtenía la entidad, la convertía a DTO con toDTO(), llamaba espacio.setActivo(false) sobre el DTO (un objeto en memoria sin relación con JPA), y nunca persistía el cambio en la base de datos. Fix Se reescribió delete() para operar sobre la entidad JPA directamente y llamar a espaciosRepository.save(espacio). 
- EspacioResponseDTO espacio = toDTO(espaciosRepository.findById(id).get());
- espacio.setActivo(false);
+ Espacio espacio = espaciosRepository.findById(id)
+     .orElseThrow(() -> new ResourceNotFoundException("Espacio no encontrado: " + id));
+ espacio.setActivo(false);
+ espaciosRepository.save(espacio);
Nota: este bug es especialmente silencioso porque el método retornaba 204 sin error. Sin una suite de integración que verifique el estado posterior al delete, pasa desapercibido indefinidamente.
B-06

Colección envía JSON body al endpoint de disponibilidad que espera @RequestParam

Colección Postman
Síntoma PUT /api/espacios/{id}/disponibilidad → 500 al enviar {"disponible": false} como cuerpo JSON. Causa raíz El controlador declara el parámetro como @RequestParam Boolean disponible (query string), no como @RequestBody. Jackson intentaba deserializar el body y fallaba. Fix Se corrigió la URL en la colección a /api/espacios/{id}/disponibilidad?disponible=false con body vacío.
B-07

PermisoCreateDTO usa descripcion que no existe en la entidad Permiso

security-service
Síntoma POST /api/permisos → 500. La entidad Permiso tiene columnas recurso NOT NULL y accion NOT NULL que el servicio nunca poblaba. Causa raíz El DTO PermisoCreateDTO solo tenía los campos nombre y descripcion. La entidad real no tiene descripcion; tiene recurso y accion. El servicio llamaba permiso.setNombre() y nada más. MySQL rechazaba el insert por las columnas NOT NULL sin valor. Archivos modificados PermisoCreateDTO · PermisoResponseDTO · PermisoService.create() · PermisoService.update() · PermisoService.toResponse()

+ return new PermisoResponseDTO(

Error en cascada: el cambio en PermisoResponseDTO (de 2 a 4 campos) rompió la compilación del servicio porque toResponse() seguía usando el constructor de 2 argumentos. Se detectó con mvn clean package y se corrigió en el mismo ciclo.
B-08

MetodoPagoCreateDTO no incluye campos obligatorios de la entidad MetodoPago

ms-pagos
Síntoma POST /api/metodos-pago → 500. Causa raíz El DTO original solo declaraba nombre, idBanco e idTipoTarjeta. La entidad MetodoPago tiene además idClienteRef NOT NULL, ultimos4 NOT NULL, mesVencimiento NOT NULL y anioVencimiento NOT NULL. MySQL rechazaba el insert. Fix Se agregaron los 4 campos faltantes al DTO. En el servicio se asignan con defaults seguros cuando el cliente no los envía: idClienteRef → 0L, ultimos4 → "0000", mesVencimiento → 1, anioVencimiento → 2099.
B-09

TipoClienteResponseDTO usa nombre de campo incorrecto — NullPointerException al calcular cobro

ms-pagos ← user-service
Síntoma POST /api/cobros → 500 con NullPointerException en PagoService. Causa raíz Desajuste entre servicios: el campo en user-service se llama DescuentoPct (capital D), por lo que Jackson lo serializa como clave JSON descuentoPct. El DTO local de ms-pagos tenía el campo nombrado descuentoPorcentaje (tipo Double). Jackson no podía mapear la clave JSON al campo del DTO → el campo quedaba null. Luego BigDecimal.valueOf(null) lanzaba NPE. Fix Se renombró el campo en el DTO local de ms-pagos a descuentoPct y se cambió el tipo a BigDecimal (consistente con user-service). En PagoService.resolverDescuentoCliente() se agregó un null-guard. 
// ms-pagos: TipoClienteResponseDTO.java
- private Double descuentoPorcentaje;
+ private java.math.BigDecimal descuentoPct;

// ms-pagos: PagoService.resolverDescuentoCliente()
- BigDecimal desc = BigDecimal.valueOf(cliente.getTipoCliente().getDescuentoPorcentaje());
+ if (cliente.getTipoCliente() == null || cliente.getTipoCliente().getDescuentoPct() == null) {
+     return BigDecimal.ZERO;
+ }
+ BigDecimal desc = cliente.getTipoCliente().getDescuentoPct();
Regla general: cuando un microservicio consume datos de otro via Feign, el DTO local debe tener exactamente el mismo nombre de campo (respetando mayúsculas) que la clave JSON que el servicio origen produce. Un mismatch silencioso de Jackson es difícil de detectar sin pruebas de integración.
B-10

DELETE Permiso bloqueado por FK de rol_permiso — violación de integridad referencial

security-service
Síntoma DELETE /api/permisos/{id} → 500 cuando el permiso tiene asignaciones en la tabla rol_permiso. Causa raíz La entidad RolPermiso referencia a Permiso con un @ManyToOne @JoinColumn real, lo que genera una FK en MySQL. El test #66 asignaba el permiso recién creado a un rol, y luego el test #68 intentaba borrar ese permiso. MySQL rechazaba el DELETE por la FK constraint. Fix Se agregó findAllByPermisoId(Long) a RolPermisoRepository. En PermisoService.delete() se borran primero todas las filas rol_permiso que referencian el permiso, y luego se borra el permiso. 
// RolPermisoRepository.java
+ List<RolPermiso> findAllByPermisoId(Long permisoId);

// PermisoService.delete()
+ rolPermisoRepository.deleteAll(rolPermisoRepository.findAllByPermisoId(id));
  permisoRepository.deleteById(id);
B-11

Colección no es idempotente — segunda ejecución falla por datos residuales

Colección Postman
Síntoma En la segunda ejecución de Newman, el test #63 POST crear permiso TEST_EXPORT recibía 422 ("Ya existe un permiso con nombre=TEST_EXPORT") porque la primera ejecución lo había creado y el cleanup (#68) había fallado (B-10 aún no estaba corregido). Todos los tests de Fase 11 que dependían del {{permisoId}} rompían en cascada. Causa raíz La variable permisoId se captura en el test-script del #63 (if (pm.response.code === 201) { pm.collectionVariables.set(...) }). Si #63 devuelve 422, la variable no se actualiza y los requests posteriores usan el ID de la ejecución anterior (que puede no existir ya o ser incorrecto). Fix Se agregó un pre-request script al request #63 que: (1) hace un GET a /api/permisos, (2) busca en la lista si ya existe un permiso con nombre = "TEST_EXPORT", (3) si lo encuentra, hace un DELETE para limpiarlo antes de que el request principal lo intente crear de nuevo. Esto hace la suite completamente idempotente en cualquier número de re-ejecuciones. 
// Pre-request de #63 — POST crear permiso TEST_EXPORT
pm.sendRequest({
    url: pm.variables.get('base') + '/api/permisos',
    method: 'GET',
    header: [{ key: 'Authorization', value: 'Bearer ' + pm.collectionVariables.get('token') }]
}, function(err, res) {
    if (!err && res.code === 200) {
        var existing = res.json().find(function(p) { return p.nombre === 'TEST_EXPORT'; });
        if (existing) {
            pm.sendRequest({
                url: pm.variables.get('base') + '/api/permisos/' + existing.id,
                method: 'DELETE',
                header: [{ key: 'Authorization', value: 'Bearer ' + pm.collectionVariables.get('token') }]
            }, function() {});
        }
    }
});
Patrón aplicado: otros tests que crean datos únicos (patente de vehículo, número de espacio) usaban ya este patrón con Date.now() como sufijo para evitar colisiones. TEST_EXPORT requería cleanup explícito porque su nombre es fijo.

7. Lecciones aprendidas

Feign y métodos HTTP

El cliente Feign por defecto (java.net.HttpURLConnection) no soporta PATCH. Existen dos soluciones: agregar Apache HC5 como cliente HTTP alternativo, o — como se hizo aquí por consistencia con el currículo — usar PUT para todas las actualizaciones parciales. Si en el futuro se necesita REST semántico estricto, se puede agregar la dependencia feign-hc5.

DTO locales en arquitectura Feign

Cuando un microservicio consume datos de otro via Feign, los DTOs locales deben coincidir exactamente con las claves JSON que el servicio origen produce. Jackson falla silenciosamente cuando el nombre no coincide (deja el campo en null), lo que genera NPEs en operaciones posteriores. La convención de nombres Lombok/Java (getDescuentoPct() → clave descuentoPct) debe ser tenida en cuenta al nombrar campos en DTOs de consumo.

Spring Data JPA: métodos únicos vs. colecciones

findBy[Campo]() que retorna Optional<T> lanza IncorrectResultSizeDataAccessException si hay más de una fila. Cuando la semántica es "dame cualquiera que cumpla la condición", siempre usar findFirstBy[Campo]() o findTopBy[Campo]().

Soft-delete: siempre operar sobre la entidad JPA

Un soft-delete que llama a setActivo(false) sobre un DTO o un objeto no gestionado por el contexto de persistencia de JPA no produce ningún efecto en la base de datos. La operación debe hacerse sobre el objeto recuperado con el repositorio (que sí está bajo gestión de JPA) y luego llamar a save() o confiar en que la transacción lo detecte como dirty.

FK entre tablas del mismo microservicio

Una FK real en la base de datos (no solo lógica) bloquea el DELETE del padre si existen hijos. Las opciones son: (1) cascade delete en la entidad JPA (CascadeType.REMOVE / orphanRemoval), (2) borrar los hijos manualmente en el servicio antes de borrar el padre, o (3) usar ON DELETE CASCADE en el DDL. Se eligió la opción 2 para mantener control explícito y poder auditar qué se borra.

Idempotencia de suites de prueba

Las suites que crean y limpian datos deben poder ejecutarse múltiples veces sin estado previo. El patrón correcto es colocar el cleanup antes de la creación (pre-request o un test de setup dedicado), no solo al final como teardown — porque si el teardown falla, la siguiente ejecución queda en estado inconsistente.
