P0 — Startup-Breaking Todos resueltos

Cualquiera de estos errores impide arrancar con ddl-auto=validate. Con ddl-auto=update el servicio arranca pero opera de forma incorrecta. Los 4 issues P0 están corregidos en el código.

ms-accesos · @Table nombre incorrecto P0 RESUELTO

La entidad Acceso.java mapea a la tabla accesos (plural), pero el DDL crea la tabla acceso (singular). Hibernate no encontrará la tabla y lanzará SchemaManagementException. Con update, creará una tabla vacía paralela.

ms-accesos/…/model/Acceso.java 
// ANTES — incorrecto
@Table(name = "accesos")

// DESPUÉS — correcto (singular, igual que el DDL)
@Table(name = "acceso")

ms-espacios · TipoEspacio.factorPrecio es Double P0 RESUELTO

El DDL define DECIMAL(5,2) pero la entidad usa Double. Hibernate mapea Double a DOUBLE (tipo flotante), que tiene código JDBC distinto a DECIMAL → el validate falla. Compara con TipoVehiculo.factorPrecio que ya usa BigDecimal correctamente.

ms-espacios/…/model/TipoEspacio.java 
import java.math.BigDecimal;

// ANTES
@Column(name = "factor_precio", nullable = false)
private Double factorPrecio;

// DESPUÉS
@Column(name = "factor_precio", nullable = false, precision = 5, scale = 2)
private BigDecimal factorPrecio;
Impacto secundario — resuelto PagoService fue refactorizado: ahora convierte el Float del DTO a BigDecimal con BigDecimal.valueOf(factor.doubleValue()) en resolverFactorEspacio(). No hay cast directo Float→BigDecimal.

user-service · TipoCliente.descuentoPct precision incorrecta P0 RESUELTO

DDL: DECIMAL(5,2). Entidad declara precision=10, scale=2. Hibernate validate compara estos metadatos y falla.

user-service/…/model/TipoCliente.java 
// ANTES
@Column(name = "descuento_pct", precision = 10, scale = 2)

// DESPUÉS
@Column(name = "descuento_pct", nullable = false, precision = 5, scale = 2)

P1 — Contrato Roto Todos resueltos

No bloquean el arranque pero producen errores en runtime, violan reglas de negocio o crean constraints incorrectos en la base de datos. Los 6 issues P1 están corregidos.

ms-pagos · Cobro.montoBase es Float (monetario) P1 RESUELTO

El DDL define monto_base DECIMAL(10,2). La entidad usa Float, un tipo de punto flotante de 32 bits (~7 dígitos significativos). Para montos monetarios esto introduce errores de redondeo inevitables. Además, Hibernate puede mapear Float a FLOAT en lugar de DECIMAL, lo que puede romper el validate.

Mismo problema con montoFinal que es Double.

ms-pagos/…/model/Cobro.java 
import java.math.BigDecimal;

// ANTES
@Column(nullable = false)
private Float montoBase;

@Column(nullable = false)
private Double montoFinal;

// DESPUÉS
@Column(name = "monto_base", nullable = false, precision = 10, scale = 2)
private BigDecimal montoBase;

@Column(name = "monto_final", nullable = false, precision = 10, scale = 2)
private BigDecimal montoFinal;
Cascada en PagoService — resuelta PagoService fue refactorizado en 10 métodos privados. Toda la pipeline de cálculo usa BigDecimal. El bug de división entera (minutos/60) también fue corregido usando BigDecimal.divide(..., 4, HALF_UP).

ms-pagos · Cobro.idAccesoRef falta UNIQUE P1 RESUELTO

DDL: id_acceso_ref BIGINT NOT NULL UNIQUE. La entidad no tiene unique=true. La regla de negocio "solo 1 cobro por acceso" está implementada en PagoService con un findByIdAcceso().isPresent(), pero sin el constraint en la entidad, no hay garantía a nivel de base de datos.

ms-pagos/…/model/Cobro.java 
// ANTES
@Column(nullable = false, name = "id_acceso_ref")

// DESPUÉS
@Column(nullable = false, name = "id_acceso_ref", unique = true)

ms-pagos · Banco.nombre tiene UNIQUE espurio P1 FALSO POSITIVO

El DDL no define UNIQUE en banco.nombre. La entidad en realidad tiene unique=true en codigo (no en nombre), lo que coincide exactamente con el DDL (codigo VARCHAR(20) NOT NULL UNIQUE). No había ningún bug aquí.

ms-pagos/…/model/Banco.java 
// ANTES
@Column(nullable = false, unique = true)
private String nombre;

// DESPUÉS
@Column(nullable = false, length = 100)
private String nombre;

ms-pagos · TipoTarjeta.nombre tiene UNIQUE espurio P1 RESUELTO

El DDL no define UNIQUE en tipo_tarjeta.nombre. El seed data ya contiene dos filas con nombre = 'Crédito' (VISA y MASTERCARD). Si Hibernate crea el índice UNIQUE, esos inserts fallan.

ms-pagos/…/model/TipoTarjeta.java 
// ANTES
@Column(nullable = false, unique = true)
private String nombre;

// DESPUÉS
@Column(nullable = false, length = 100)
private String nombre;

ms-reservas · fechaFin y fechaCreacion nullability P1 RESUELTO

fecha_fin es NOT NULL en el DDL pero la entidad no tiene @Column(nullable=false). fecha_creacion es NOT NULL DEFAULT CURRENT_TIMESTAMP en el DDL pero la entidad la marca como nullable=true.

ms-reservas/…/model/Reserva.java 
// fecha_fin — ANTES: sin @Column
private LocalDateTime fechaFin;

// DESPUÉS
@Column(nullable = false)
private LocalDateTime fechaFin;

// fecha_creacion — ANTES
@Column(nullable = true, ...)

// DESPUÉS
@Column(nullable = false, updatable = false)

auth-service · UserCredential.activo sin valor por defecto P1 RESUELTO

El DDL define activo TINYINT(1) NOT NULL DEFAULT 1. La entidad tiene el campo como Boolean activo sin inicializar. Hibernate siempre incluye todas las columnas mapeadas en el INSERT — el DEFAULT del DDL nunca se activa. Si algún código crea un UserCredential sin llamar a setActivo(true), el INSERT falla con constraint violation.

auth-service/…/model/UserCredential.java 
// ANTES
@Column(nullable = false)
private Boolean activo;

// DESPUÉS
@Column(nullable = false)
private Boolean activo = true;
