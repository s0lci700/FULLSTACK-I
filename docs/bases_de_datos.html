Bases de Datos

Documentación del modelo de datos del Sistema de Estacionamiento Inteligente. Diseñado para leerse tanto sin experiencia técnica como con ella.

Mostrar nivel:
Vista principiante activa
💡

¿Qué es una base de datos? Imagina una hoja de cálculo muy ordenada que vive en el servidor. Cada "tabla" es una pestaña con filas y columnas, y ahí se guarda toda la información del sistema: usuarios, vehículos, pagos, etc.

🧩

¿Por qué hay 9 bases de datos y no una sola? Este sistema usa microservicios: en lugar de un programa gigante, son 9 programas pequeños que se comunican entre sí. Cada uno tiene su propia base de datos para que pueda funcionar de manera independiente. Si falla el sistema de pagos, los usuarios igual pueden hacer reservas.

db_auth

Puerto 8081 auth-service

Guarda quién puede entrar al sistema. Cuando alguien inicia sesión con email y contraseña, este servicio verifica si las credenciales son correctas y, si lo son, entrega un token JWT — una especie de pase digital que demuestra identidad sin necesidad de volver a ingresar la contraseña en cada petición.

Analogía: Es como el guardia de seguridad que revisa tu carnet en la entrada. Una vez que te identifica, te da un brazalete (token) que te permite moverse libremente por el resto del edificio sin que te vuelvan a pedir el carnet en cada puerta.

Tablas

rol

Define los tipos de usuario del sistema: ADMIN, USER, OPERATOR. Cada rol tiene distintos permisos.

user_credential

Almacena el email y la contraseña cifrada de cada usuario. Apunta a su rol y al cliente que representa.

🔒

Las contraseñas se almacenan como hash BCrypt con strength 12. Nunca se guarda la contraseña en texto plano. El campo id_cliente_ref es una FK lógica hacia db_usuarios.cliente — no hay constraint real porque cruza bases de datos.

db_seguridad

Puerto 8083 security-service

Si db_auth decide "eres quien dices ser", db_seguridad decide "¿qué puedes hacer?". Define una lista de acciones permitidas (permisos) y las asigna a cada rol.

Analogía: El brazalete del guardia (db_auth) te permite entrar. db_seguridad es la lista de qué zonas puede acceder cada color de brazalete. El rojo entra a todo, el verde solo a la cafetería.

Tablas

permiso

Define cada acción posible en el sistema. Ej: "VER_VEHICULOS en GET /api/vehiculos".

rol_permiso

Tabla puente que asigna permisos a roles. Un rol puede tener muchos permisos; un permiso puede estar en varios roles.

🔗

id_rol_ref en rol_permiso es una FK lógica hacia db_auth.rol. No hay constraint de base de datos porque son dos schemas distintos; la integridad se mantiene a nivel de aplicación.

db_usuarios

Puerto 8082 user-service

Guarda el perfil completo de los clientes del estacionamiento: nombre, RUT, contacto, tipo de cliente y suscripciones activas. Es diferente a db_auth, que solo guarda el email/contraseña — esta BD guarda quién es la persona.

Analogía: db_auth es la tarjeta de acceso. db_usuarios es la ficha del cliente en el archivo del negocio, con todos sus datos personales y qué plan tiene contratado.

Tablas

tipo_cliente

Categorías de cliente: ESTANDAR (0% desc), FRECUENTE (10% desc), VIP (20% desc).

cliente

Datos del cliente: RUT, nombre, email, teléfono y su categoría.

suscripcion

Planes de suscripción disponibles (BASICA 5%, PREMIUM 15%, ANUAL 25%). Cada uno otorga un descuento adicional en los cobros.

cliente_suscripcion

Registra qué cliente tiene contratado qué plan, con fechas de inicio y fin.

db_vehiculos

Puerto 8084 ms-vehiculos

Registra los vehículos asociados a cada cliente. Cuando un vehículo entra al estacionamiento, el sistema escanea la patente y busca aquí para saber si está registrado y a quién pertenece.

Analogía: Es el libro de registros del estacionamiento donde están anotadas todas las patentes con el nombre de su dueño y el tipo de vehículo.

⚠️

Estado de implementación: Este servicio está parcialmente implementado (~40%). Tiene las entidades y un endpoint de validación de patente. Faltan: CRUD completo, DTOs, excepciones globales.

Tablas

tipo_vehiculo

Categorías: AUTO (×1.0), MOTO (×0.7), CAMIONETA (×1.5), BUS (×2.0). El factor multiplica el precio del cobro.

vehiculo

Datos del vehículo: patente única, marca, modelo, color, año, tipo y a qué cliente pertenece.

💻

Nombres de tabla generados por SpringPhysicalNamingStrategy: clase TipoVehiculo → tabla tipo_vehiculo, clase Vehiculo → tabla vehiculo. El campo Java factor_precio es Float en la entidad — la columna usa DECIMAL(5,2), compatible con Hibernate sin errores de validación.

db_espacios

Puerto 8085 ms-espacios

Administra los lugares físicos del estacionamiento. Cada espacio tiene un número único, está en una zona y piso, y en todo momento se sabe si está disponible o no. Cuando un vehículo entra, el espacio se marca como ocupado; cuando sale, vuelve a estar disponible.

Analogía: Es el tablero digital con luces rojas y verdes que ves al entrar a un estacionamiento de mall. Verde = libre, Rojo = ocupado.

Tablas

tipo_espacios

Categorías: ESTANDAR (×1.0), DISCAPACITADO (×1.0), MOTOCICLETA (×0.5), GRANDE (×1.5).

espacios

Cada espacio físico: número (Ej: A-01), zona, piso, tipo, y si está disponible o no en este momento.

🐛

Bug conocido en la entidad: Espacios.java le falta la relación @ManyToOne TipoEspacios. La columna id_tipo_espacio ya existe en la tabla SQL para cuando se corrija. Los campos disponible y activo son Integer en Java (deberían ser Boolean), por eso la columna usa INT en lugar de TINYINT(1).

db_tarifas

Puerto 8088 ms-tarifas

Define cuánto cuesta estacionar según el horario. No es un precio fijo: hay un precio base por hora, pero se aplican multiplicadores según si es día de semana, fin de semana, o festivo, y según el bloque horario (mañana, tarde, noche).

Analogía: Como el precio del taxi que sube en hora punta o los fines de semana. La tarifa base es el "bajada de bandera" y los horarios ajustan el costo según cuándo estacionas.

Tablas

tarifa

Define el precio base por hora. Ej: TARIFA_BASE = $1.500/hora.

horario_tarifa

Rangos horarios con multiplicadores. Ej: fin de semana ×1.5 → $2.250/hora.

db_reservas

Puerto 8086 ms-reservas

Permite a los clientes reservar un espacio con anticipación para una fecha y hora determinada. Una reserva no ocupa físicamente el espacio, pero garantiza que habrá uno disponible cuando el cliente llegue.

Analogía: Como reservar mesa en un restaurante. Llamas con anticipación, te guardan la mesa, y cuando llegas ya saben dónde sentarte sin tener que buscar espacio disponible.

Tablas

reserva

Registra cada reserva con cliente, vehículo, tipo de espacio deseado, fechas y estado (PENDIENTE / CONFIRMADA / CANCELADA / EXPIRADA).

🔗

Este servicio necesita comunicarse con ms-vehiculos (para verificar que la patente existe), ms-espacios (para confirmar disponibilidad), y user-service (para verificar el cliente). Esa comunicación se hace con Feign Client — pendiente implementar.

🗝️

Todas las referencias a otras BDs son FK lógicas: id_cliente_ref, id_vehiculo_ref e id_tipo_espacio_ref son simples BIGINT sin constraint. La validación se hace en el servicio antes de insertar.

db_accesos

Puerto 8087 ms-accesos

Es el registro de entrada y salida de cada vehículo. Cuando un vehículo entra, se crea un acceso con hora de entrada y se marca el espacio como ocupado. Cuando sale, se registra la hora de salida y el espacio queda libre nuevamente.

Analogía: El tique de estacionamiento. El tique tiene el número del espacio asignado, la hora de entrada (y la de salida cuando pagas). Una vez pagado y salido, ese espacio vuelve a estar disponible.

Tablas

acceso

Un registro por cada entrada al estacionamiento. Tiene la patente escaneada, el espacio asignado, hora de entrada, hora de salida (null si aún está adentro), y estado (ACTIVO / COMPLETADO).

db_pagos

Puerto 8089 ms-pagos

Gestiona el cobro al momento de la salida. Calcula el monto final aplicando la tarifa del horario, el tipo de vehículo, el tipo de espacio, y los descuentos del cliente. Guarda también los métodos de pago (tarjetas) registrados por cada cliente.

Analogía: La caja del estacionamiento. Recibe el tique (acceso), calcula cuánto debes según el tiempo y hora, aplica descuentos si tienes tarjeta del cliente, y procesa el cobro a tu tarjeta registrada.

Tablas

banco

Bancos con descuentos por convenio. Ej: Banco Estado otorga 5% adicional.

tipo_tarjeta

Categorías de tarjeta: Crédito VISA, Débito Mastercard, etc.

metodo_pago

Tarjetas registradas por cada cliente. Guarda solo los últimos 4 dígitos (nunca el número completo).

cobro

El registro final del pago: monto base, cada descuento aplicado y el monto final cobrado. Relación 1:1 con un acceso.

Fórmula de cobro (ms-pagos)

🧮

El sistema no cobra un precio fijo. El monto final se calcula combinando información de cuatro bases de datos distintas. Por eso ms-pagos depende de casi todos los otros servicios.

Paso 1 — Monto base

monto_base = precio_base_hora × multiplicador_horario × factor_tipo_vehiculo × factor_tipo_espacio × (minutos / 60)
  • 1precio_base_hora → viene de db_tarifas.tarifa. Ej: $1.500/hora.
  • 2multiplicador_horario → viene de db_tarifas.horario_tarifa según el día y hora de entrada. Ej: fin de semana = ×1.5.
  • 3factor_tipo_vehiculo → viene de db_vehiculos.tipo_vehiculo. Ej: MOTO = ×0.7 (más barato), BUS = ×2.0 (más caro).
  • 4factor_tipo_espacio → viene de db_espacios.tipo_espacios. Ej: espacio GRANDE = ×1.5.
  • 5minutos / 60 → tiempo estacionado, calculado de db_accesos.acceso (salida - entrada).

Paso 2 — Descuentos (se aplican en cascada)

monto_final = monto_base × (1 − desc_tipo_cliente / 100) × (1 − desc_suscripcion / 100) × (1 − desc_banco / 100)
  • Adesc_tipo_cliente → viene de db_usuarios.tipo_cliente. Ej: FRECUENTE = 10%.
  • Bdesc_suscripcion → viene de db_usuarios.suscripcion si el cliente tiene plan activo. Ej: PREMIUM = 15%.
  • Cdesc_banco → viene de db_pagos.banco según la tarjeta usada. Ej: Banco Estado = 5%.
📊

Ejemplo concreto: María (ESTANDAR, sin suscripción, Banco Estado) estaciona su AUTO en espacio ESTANDAR un martes a las 9:00 y sale a las 11:30 (150 min).

monto_base = 1500 × 1.00 × 1.00 × 1.00 × (150/60) = $3.750
monto_final = 3750 × (1-0) × (1-0) × (1-0.05) = $3.562,50

FK lógicas vs FK reales

🔗

Una FK (Foreign Key) es una columna que apunta al ID de otra tabla para relacionarlas. En una base de datos normal, MySQL puede validar automáticamente que ese ID exista. Pero cuando las dos tablas están en bases de datos distintas, MySQL no puede hacer esa validación. A esa referencia "sin constraint" se le llama FK lógica.

Convención de colores en esta documentación

id Primary Key — identificador único de la fila
id_xxx Foreign Key real — MySQL valida la existencia
id_xxx_ref FK lógica — referencia cross-BD, validada en código
campo Unique — no permite valores duplicados
