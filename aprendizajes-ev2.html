@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(NotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(NotFoundException ex) {
        return ResponseEntity
            .status(HttpStatus.NOT_FOUND)           // 404
            .body(new ErrorResponse("NOT_FOUND", ex.getMessage()));
    }

    @ExceptionHandler(ConflictException.class)
    public ResponseEntity<ErrorResponse> handleConflict(ConflictException ex) {
        return ResponseEntity
            .status(HttpStatus.CONFLICT)             // 409
            .body(new ErrorResponse("CONFLICT", ex.getMessage()));
    }

    @ExceptionHandler(BadRequestException.class)
    public ResponseEntity<ErrorResponse> handleBadRequest(BadRequestException ex) {
        return ResponseEntity
            .status(HttpStatus.BAD_REQUEST)          // 400
            .body(new ErrorResponse("BAD_REQUEST", ex.getMessage()));
    }

    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<ErrorResponse> handleBusiness(BusinessException ex) {
        return ResponseEntity
            .status(HttpStatus.UNPROCESSABLE_ENTITY)  // 422
            .body(new ErrorResponse("BUSINESS_RULE_VIOLATION", ex.getMessage()));
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ValidationErrorResponse> handleValidation(
            MethodArgumentNotValidException ex) {
        Map<String, String> campos = ex.getBindingResult()
            .getFieldErrors().stream()
            .collect(Collectors.toMap(
                FieldError::getField,
                FieldError::getDefaultMessage));
        return ResponseEntity
            .status(HttpStatus.BAD_REQUEST)          // 400
            .body(new ValidationErrorResponse("VALIDATION_FAILED", campos));
    }
}

P1 Explique la arquitectura general. Indique el rol de Eureka, API Gateway, Auth Service, User Service y al menos cuatro microservicios de dominio.
Cliente HTTP→ API Gateway :8080→ JwtAuthFilter→ Microservicio lb://→ Eureka :8761
  • Eureka Server (:8761): Registro de servicios. Cada microservicio se registra al arrancar con su spring.application.name. Actúa como DNS interno — cuando Gateway necesita enviar una request a lb://ms-reservas, Eureka resuelve qué IP y puerto tiene ese servicio activo.
  • API Gateway (:8080): Punto único de entrada. Todas las requests externas llegan aquí. Define rutas lb:// en application.yaml. El filtro JwtAuthFilter valida el token JWT antes de dejar pasar la request.
  • Auth Service (:8081): Único emisor de tokens JWT. Recibe email + password en POST /api/auth/login, valida con BCrypt strength 12, y retorna un JWT firmado válido por 24h. Los demás servicios NO emiten tokens.
  • User Service (:8082): Gestiona clientes y suscripciones. Entidades: Cliente, TipoCliente, Suscripcion, ClienteSuscripcion. ms-pagos lo consulta via Feign para obtener descuentos por tipo de cliente y suscripción.
  • ms-reservas (:8086): Ciclo de vida de reservas. Estados: PENDIENTE → CONFIRMADA → FINALIZADA (o CANCELADA). Al crear, valida cliente+vehículo+espacio via Feign.
  • ms-accesos (:8087): Registra entrada (estado ACTIVO, marca espacio ocupado) y salida (estado COMPLETADO, calcula minutos, libera espacio).
  • ms-pagos (:8089): Calcula y persiste cobros. Consulta 7 servicios via Feign para obtener tarifa, horario, factores de vehículo/espacio y descuentos.
  • ms-reportes (:8090): Sin base propia. Agrega datos de accesos, cobros y espacios via Feign y genera reportes no persistentes.

        .orElseThrow(() -> new NotFoundException("Reserva no encontrada"));

P5 ¿Por qué el proyecto usa referencias lógicas Long entre microservicios en vez de relaciones JPA directas?
ℹ
Esta es la pregunta más conceptual — hay que entenderla profundo.
  • Cada microservicio tiene su propia base de datos. ms-reservas tiene db_reservas, ms-vehiculos tiene db_vehiculos. Son instancias MySQL completamente separadas.
  • JPA @ManyToOne requiere que ambas entidades estén en el mismo datasource. No se puede hacer @ManyToOne de Reserva a Vehiculo si están en bases distintas — JPA no puede hacer JOIN entre bases diferentes.
  • Solución: guardar solo el ID como Long idVehiculoRef. Cuando se necesita el vehículo, se consulta via Feign: vehiculoClient.findById(idVehiculoRef).
  • Beneficio: cada servicio puede escalar, desplegarse y mantenerse independientemente. Si ms-vehiculos cae, ms-reservas solo falla en create() pero puede seguir atendiendo otras requests.
  • Compromiso: la integridad referencial es responsabilidad del código (validaciones Feign), no de la base de datos (no hay FOREIGN KEY entre bases).
// MAL — no se puede hacer esto entre bases distintas:
@ManyToOne
@JoinColumn(name = "id_vehiculo")
private Vehiculo vehiculo;  // Vehiculo está en db_vehiculos, Reserva en db_reservas ❌

// BIEN — referencia lógica:
@Column(name = "id_vehiculo_ref", nullable = false)
private Long idVehiculoRef;  // Solo el ID, se valida via Feign al crear reserva ✓

Guía: Excepciones y Códigos HTTP

El indicador IE 2.3.4 vale 18% y fue el punto más débil. Esta guía cubre todo lo necesario para dominarlo.

La jerarquía de excepciones del proyecto

// Todas las excepciones de negocio extienden RuntimeException
public class NotFoundException    extends RuntimeException {}  → 404
public class ConflictException     extends RuntimeException {}  → 409
public class BadRequestException   extends RuntimeException {}  → 400
public class BusinessException     extends RuntimeException {}  → 422

Formato de respuesta de error (siempre el mismo)

{
  "error":     "NOT_FOUND",
  "mensaje":   "Reserva no encontrada con id: 99",

}

// Para validaciones de Bean Validation, agrega:
{

Principio clave: los controllers no usan try/catch

✓
Los controllers solo reciben HTTP y llaman al service. Las excepciones burbujean automáticamente hasta el GlobalExceptionHandler. Esto mantiene los controllers limpios y la lógica de error en un solo lugar.

Guía: Errores Feign entre microservicios

Cuando un microservicio llama a otro y el otro falla, se genera una FeignException. Hay que saber interpretarla.

// Feign lanza diferentes subclases según el código HTTP:
FeignException.NotFound         → el servicio remoto retornó 404
FeignException.BadRequest       → el servicio remoto retornó 400
FeignException.Conflict         → el servicio remoto retornó 409
FeignException.ServiceUnavailable → el servicio remoto retornó 503
RetryableException              → el servicio remoto está caído (conexión rechazada)

// Patrón de manejo en el service:
try {
    EspacioDTO espacio = espacioClient.findById(idEspacio);
} catch (FeignException.NotFound e) {
    log.error("Espacio {} no encontrado en ms-espacios", idEspacio);
    throw new NotFoundException("Espacio no encontrado: " + idEspacio);
} catch (FeignException e) {
    log.error("Error comunicando con ms-espacios: {}", e.getMessage());
    throw new BusinessException("Servicio de espacios no disponible");
}
⚠
Error común: usar null check en vez de catch FeignException
Feign lanza excepción cuando el servicio retorna 404 — no retorna null. Un if (espacio == null) nunca se activa. Siempre usar try/catch.

Cómo documentar tu aporte en una defensa

Estructura en 4 partes:

2. Funcionalidades específicas que implementaste

Lo que nunca decir

  • "Me encargué de la documentación y validaciones" (sin especificar cuáles)
  • "Trabajé en varios servicios" (sin nombrarlos)
  • "Hicimos todo juntas" (el profesor sabe que no — tiene los commits)

Lo que siempre decir

  • Nombres de microservicios específicos: "lideré ms-accesos con X commits"
  • Métodos concretos: "implementé registrarEntrada() y registrarSalida() en AccesoService"
  • Artefactos: "escribí manage.ps1 con X funciones: start, stop, restart, db"
  • Resultado verificable: "la colección Newman pasa 74/74 con mis tests de idempotencia"
